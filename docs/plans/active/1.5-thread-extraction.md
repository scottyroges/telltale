# Plan 1.5: Thread Extraction

**Status:** Planned
**Depends on:** 1.2 (needs conversation service), 1.4 (needs working conversations to test against)
**Blocks:** 1.6

## Goal

After each assistant response, extract structured threads (people, places, events, emotions, unexplored details) via a separate Claude call. Persist as StoryThread records. Threads power follow-up question intelligence and cross-session context.

## Manual Steps (Before Code)

None.

## Key Steps

- Create `src/services/extraction-prompt.ts`:
  - Exports `THREAD_EXTRACTION_PROMPT` as a string constant
  - Instructs Claude to analyze the conversation and extract structured threads
  - Requests JSON output in a specific format:
    ```json
    {
      "threads": [
        { "type": "ENTITY", "content": "sister Maria — mentioned growing up together in Calabria" },
        { "type": "EVENT", "content": "the 1978 flood that destroyed the family farm" },
        { "type": "EMOTION", "content": "pride when talking about first job at the bakery" },
        { "type": "DETAIL", "content": "mentioned a 'secret recipe' but didn't elaborate" }
      ]
    }
    ```
  - Instructions: be specific (include names, dates, places), note emotional tone, flag details mentioned but not explored, avoid duplicating threads that have already been extracted
- Create `src/services/thread-extraction.service.ts`:
  - **`extractThreads(sessionId: string, recentMessages: Message[])`**
    - Loads existing threads for the session via `storyThreadRepository.findBySessionId()`
    - Builds prompt: extraction system prompt + recent messages + list of already-extracted threads (to avoid duplicates)
    - Calls `anthropic.messages.create()` (NOT streaming — this is a structured extraction, not user-facing)
    - Parses JSON response, validates structure
    - Filters out threads that are too similar to existing ones (simple string similarity or exact match on content)
    - Creates new threads via `storyThreadRepository.createMany()`
    - Returns the newly created threads
  - **`getThreads(sessionId: string)`**
    - Returns all threads for a session via `storyThreadRepository.findBySessionId()`
- Integrate into `src/services/conversation.service.ts`:
  - After `sendMessage` returns and the assistant message is persisted, fire thread extraction
  - **Fire-and-forget**: thread extraction runs asynchronously and does NOT block the response to the client. The response is already returned when extraction starts.
  - Use a simple `extractThreads(...).catch(console.error)` pattern — if extraction fails, the conversation is unaffected
  - Pass the last N messages (e.g., last 6 messages — 3 exchanges) as context for extraction, not the entire history
- Add `getThreads` to the session tRPC router:
  - `getThreads` — protected query. Input: `{ sessionId: string }`. Returns threads for the session. Verifies ownership.
- Write unit tests:
  - `tests/services/thread-extraction.service.test.ts`:
    - Mock Anthropic client (return structured JSON)
    - Mock storyThreadRepository
    - Test: extracts threads from messages, deduplicates against existing, creates new records
    - Test: handles malformed JSON gracefully (doesn't throw)
    - Test: handles empty extraction result
    - Test: passes existing threads to prompt for deduplication context

## Notes

- Thread extraction is **non-blocking** — it must never slow down the conversation. The user gets the response immediately, and thread extraction happens in the background.
- The extraction call uses `anthropic.messages.create()` (not streaming) because we need the full structured JSON response.
- Model for extraction: can use a smaller/faster model if desired (e.g., Haiku) since this is a structured extraction task, not a creative conversation. Start with the same model as conversation and optimize later.
- Deduplication is simple for MVP: compare extracted thread content against existing thread content. Exact or near-exact matches are skipped. Sophisticated semantic deduplication can come later.
- The extraction prompt asks Claude to be specific — "sister Maria" not "a family member", "1978 flood" not "a natural disaster". Specificity makes threads useful for follow-up questions.
- Error handling: if the Claude response isn't valid JSON, log the error and skip extraction for this turn. Don't crash the service.
- The `getThreads` tRPC procedure is added to enable a future UI that shows extracted threads (sidebar or thread panel). The UI for displaying threads is NOT part of this plan.
