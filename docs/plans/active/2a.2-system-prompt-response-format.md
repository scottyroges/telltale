# Plan 2a.2: Core Memory Block — System Prompt + Response Format

**Status:** Complete
**Created:** 2026-02-26
**Parent plan:** 2a (core memory block)
**Depends on:** 2a.1 (schema + repository layer)
**Blocks:** 2a.3 (context assembly + conversation service integration)

## Goal

Rewrite the interviewer system prompt to request a core memory block instead of insights, update the response parser to handle the new JSON shape (`updatedCoreMemory` replacing `insights[]`), and increase `maxTokens` to accommodate the larger response. After this PR, the LLM is instructed to return memory blocks and the parser can extract them — but nothing reads `updatedCoreMemory` yet. That wiring happens in 2a.3.

## Why This Is Safe as a Standalone PR

The response parser adds `updatedCoreMemory: string | null` to `ParsedResponse` and sets `insights` to `[]`. Existing callers (`conversationService`) read `text`, `insights`, `shouldComplete`, and `parsed` — they'll still work. `insights: []` means no insights get created (which is the intended behavior — we're switching away from insights). `updatedCoreMemory` is ignored until 2a.3 wires it in.

The prompt change takes effect immediately — the LLM will start returning the new JSON shape on the next conversation turn. The parser handles it. The conversation service ignores the new field and creates zero insights. This is correct transitional behavior.

**What changes** (reference parent plan 2a, "What the Core Memory Block Is" section lines 23-71 for detailed content guidance)**:**

- **Remove** the insight extraction instructions: the insight types list, specificity guidance, "don't repeat insights" rule
- **Remove** the `insights[]` JSON example
- **Add** core memory block instructions:
  - The two-section structure: `## Book Memory` and `## Interview Memory`
  - What goes in Book Memory (key people, life narrative, emotional patterns) — durable, cross-interview
  - What goes in Interview Memory (topic, current thread, active threads, session notes) — session-scoped
  - Size guideline: ~2,000-3,000 characters total, communicated as a soft suggestion
  - Guidance: most turns the book section changes little; the interview section changes frequently
  - What to do when no memory exists yet ("You have no existing memory of this subject" — create the initial block from scratch)
- **Update** the JSON example to show `{ response, updatedCoreMemory, shouldComplete }`
- **Update** the closing instruction from "note-taking" framing to "memory management" framing: the conversational response is still the priority

### 2. Response Parser (`src/services/response-parser.ts`)

**Type changes:**
- Add `updatedCoreMemory: string | null` to `ParsedResponse`
- Keep `insights: ParsedInsight[]` on `ParsedResponse` (always `[]` — removed in Phase 4)

**`PARSE_CORRECTION_PROMPT`:**
- Update example JSON to `{ "response": "...", "updatedCoreMemory": "## Book Memory\n...", "shouldComplete": false }`

**`parseInterviewerResponse()`:**
- Replace the insight validation loop (lines ~42-57) with `updatedCoreMemory` extraction:
  - If `obj.updatedCoreMemory` is a non-empty string → use it
  - Otherwise → `null` (signals "don't overwrite existing memory")
- Set `insights` to `[]` always
- Every return path (success and failure) includes `updatedCoreMemory` — failures return `null`

**Dead code:**
- Remove `VALID_TYPES` constant (only used by the removed insight validation loop, internal to this file)

**`parseWithRetry()`:**
- The two `parseInterviewerResponse` calls return the new shape automatically
- The hardcoded failure return (line ~73: `{ text: raw, insights: [], shouldComplete: false, parsed: false }`) must also include `updatedCoreMemory: null`
