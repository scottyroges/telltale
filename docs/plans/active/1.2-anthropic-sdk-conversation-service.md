# Plan 1.2: LLM Provider + Conversation Service

**Status:** Planned
**Depends on:** 1.1 (needs repositories and domain types)
**Blocks:** 1.3, 1.5, 1.6

## Goal

Define a provider-agnostic LLM interface, implement it with the Anthropic SDK, and build the core conversation service with the initial interviewer system prompt. The conversation service is the heart of the product — the service that orchestrates interview conversations — and it must have zero knowledge of which LLM provider is behind the interface.

## Manual Steps (Before Code)

- `npm install @anthropic-ai/sdk`
- Ensure `ANTHROPIC_API_KEY` is set in `.env` (already in `.env.example`)

## Key Steps

### LLM Provider Interface

- Create `src/domain/llm-provider.ts`:
  - Domain-level interface — types only, no runtime code
  - Defines the contract that any LLM provider must satisfy:
    ```typescript
    export interface LLMMessage {
      role: "user" | "assistant";
      content: string;
    }

    export interface LLMResponse {
      content: string;
    }

    export interface LLMProvider {
      generateResponse(
        systemPrompt: string,
        messages: LLMMessage[]
      ): Promise<LLMResponse>;
    }
    ```
  - `LLMMessage` uses only `user` and `assistant` roles — system prompt is a separate argument, not a message. This is the lowest common denominator across providers.
  - `LLMResponse` returns plain `content` — the service doesn't need token counts, stop reasons, or other provider-specific metadata.

### Anthropic Implementation

- Create `src/lib/anthropic.ts`:
  - Singleton Anthropic SDK client, same pattern as `src/lib/prisma.ts`
  - `import "server-only"` guard
  - Reads `ANTHROPIC_API_KEY` from environment
  - Exports `anthropic` client instance (used only by the provider implementation)
- Create `src/lib/anthropic-provider.ts`:
  - Implements `LLMProvider` using the Anthropic SDK
  - Maps `LLMMessage[]` to Anthropic's message format
  - Passes `systemPrompt` as Anthropic's `system` parameter
  - Calls `anthropic.messages.create()` with model `claude-sonnet-4-5-20250929`
  - Extracts text content from Anthropic's response and returns `{ content: string }`
  - Model name is a constant within this file — easy to change, but not externally configurable yet

### Provider Wiring

- Create `src/lib/llm.ts`:
  - `import "server-only"` guard
  - Exports `llmProvider` — a configured `LLMProvider` instance
  - Currently wires up `AnthropicProvider`; switching providers means changing this one file
  - This is the only file in the codebase that knows which provider is active

### System Prompt

- Create `src/services/prompt.ts`:
  - Exports `INTERVIEWER_SYSTEM_PROMPT` as a string constant
  - Warm, curious interviewer persona
  - Instructions: ask open-ended follow-ups, pick up on names/places/emotions, avoid yes/no questions, keep responses conversational (not too long), reference earlier details to show active listening
  - Includes role framing: "You are a skilled life story interviewer helping someone capture their personal history..."

### Conversation Service

- Create `src/services/conversation.service.ts`:
  - Imports `llmProvider` from `@/lib/llm` (the provider instance, typed as `LLMProvider`)
  - Imports repositories from `@/repositories/`
  - **`startInterview(bookId: string, questionId: string)`**
    - Looks up the question via `questionRepository.findById(questionId)` to get the prompt text
    - Creates a new Interview via `interviewRepository.create({ bookId, questionId })`
    - Generates the opening message: calls `llmProvider.generateResponse()` with the system prompt + a single user message like "The topic for this conversation is: {question prompt}. Please greet the storyteller warmly and ask an opening question about this topic."
    - Persists the assistant message via `messageRepository.create({ interviewId, role: 'ASSISTANT', content })`
    - Updates the BookQuestion status to `STARTED` via `bookQuestionRepository.updateStatus()`
    - Returns `{ interviewId, openingMessage }`
  - **`sendMessage(interviewId: string, content: string)`**
    - Persists the user message via `messageRepository.create({ interviewId, role: 'USER', content })`
    - Loads message history via `messageRepository.findByInterviewId(interviewId)`
    - Maps domain `Message[]` to `LLMMessage[]` (role + content only)
    - Calls `llmProvider.generateResponse(INTERVIEWER_SYSTEM_PROMPT, messages)`
    - Persists the assistant response via `messageRepository.create({ interviewId, role: 'ASSISTANT', content: response.content })`
    - Returns the assistant response text as a plain string
  - **`getInterviewMessages(interviewId: string)`**
    - Returns messages via `messageRepository.findByInterviewId(interviewId)`
  - **`completeInterview(interviewId: string)`**
    - Updates interview status to `COMPLETE` via `interviewRepository.updateStatus(interviewId, 'COMPLETE')`

## Notes

- The conversation service is **framework-agnostic** — no Next.js, tRPC, or React imports. It imports only from repositories, domain types, and lib.
- The conversation service has **zero Anthropic knowledge** — it calls `llmProvider.generateResponse()` and gets back a string. Switching to OpenAI, Gemini, or a local model means writing a new provider implementation and changing the wiring in `src/lib/llm.ts`. No service code changes.
- `sendMessage` returns a plain string, not a stream. Per ADR 013, the interviewer model produces short responses (a reflection + follow-up question) where streaming adds complexity without UX benefit. Streaming can be added to the `LLMProvider` interface later (e.g., `generateStreamingResponse()`) if lag becomes noticeable.
- `startInterview` returns the opening message synchronously — it's a short greeting.
- The system prompt is a separate file so it can be imported by both the conversation service (Plan 1.2) and the insight extraction service (Plan 1.5) if needed.
- The `LLMProvider` interface is intentionally minimal — just `generateResponse`. Additional methods (structured output, streaming, embeddings) can be added to the interface as needed in future plans.
- Token usage tracking is not included in this plan — can be added as a field on `LLMResponse` later as a cross-cutting concern.
- Plans 1.5 and 1.6 also call the LLM directly — they should use `llmProvider` through the same interface. Those plans need updating to match.
