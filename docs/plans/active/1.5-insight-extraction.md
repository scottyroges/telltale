# Plan 1.5: Insight Extraction

**Status:** Planned
**Depends on:** 1.1 (Insight model + repository), 1.2 (conversation service + LLM provider + system prompt)
**Blocks:** 1.6
**Decision:** ADR 014 (Tentative — Option B, inline structured output)

## Goal

Make the interviewer produce structured insights — its own "mental notes" — as part of each response. Insights capture what the AI noticed, what it wants to explore later, and what it's following up on now. They are persisted as Insight records and injected back into context on every turn, giving the interviewer a persistent notepad that survives across turns, summarization, and pauses.

This plan modifies the conversation service and system prompt from Plan 1.2 to use a structured JSON response format. The LLMProvider interface stays unchanged — it still returns `{ content: string }`, but that string is now JSON containing both the response text and insights.

## Prototype (Before Code)

Before building infrastructure, validate the approach by testing the prompt directly against the Anthropic API or console. Run 5-10 turns of a sample interview with the structured output instructions and assess:

1. Does conversation quality hold up? Is the AI still warm and natural?
2. Are the insights useful? Do they capture emotional nuance, unexplored threads, and reasoning — or just restate what the user said?

If conversation quality suffers, iterate on the prompt. If it fundamentally doesn't work, fall back to Option A from ADR 014.

## Key Steps

### Response Parser

- Create `src/services/response-parser.ts`:
  - **`parseInterviewerResponse(raw: string): { text: string, insights: ParsedInsight[] }`**
  - `ParsedInsight` is `{ type: InsightType, content: string }`
  - Attempts to parse `raw` as JSON with shape `{ response: string, insights: Array<{ type, content }> }`
  - Graceful fallback chain:
    1. Valid JSON with both fields → normal path
    2. Valid JSON with `response` but missing/empty `insights` → use response text, empty insights
    3. Invalid JSON → treat entire `raw` string as response text, empty insights array
    4. Valid JSON but individual insights have wrong structure → use response text, skip malformed entries
  - This is the seam that changes if we switch to Option A — the parser would become a no-op (pass through text, empty insights) and a separate extraction call would produce insights instead

### System Prompt Update

- Modify `src/services/prompt.ts`:
  - Add structured output instructions to `INTERVIEWER_SYSTEM_PROMPT`
  - Tell the AI to respond in JSON format:
    ```json
    {
      "response": "Your conversational response to the storyteller...",
      "insights": [
        { "type": "ENTITY", "content": "sister Maria — older, bossy dynamic, user showed warmth" },
        { "type": "DETAIL", "content": "Teresa mentioned alongside Maria but no details given — worth exploring" },
        { "type": "EMOTION", "content": "pride and nostalgia when describing dad's hardware store" }
      ]
    }
    ```
  - Insight instructions:
    - Be specific — "sister Maria" not "a family member", "1978 flood" not "a natural disaster"
    - Capture emotional tone and why it matters — "pride when talking about dad" not just "dad mentioned"
    - Flag things the user mentioned but didn't elaborate on — these are future probes
    - Note what you're actively following up on vs. saving for later
    - Don't repeat insights that are already in your notes — build on them or update them
  - Insight types: `ENTITY` (person, place, thing), `EVENT` (something that happened), `EMOTION` (emotional moment), `DETAIL` (unexplored detail worth probing)
  - Emphasize: the conversational response is the priority. Insights are secondary notes. Never let note-taking make the conversation feel mechanical.

### Conversation Service Updates

- Modify `src/services/conversation.service.ts`:
  - **`sendMessage(interviewId, content)`** changes:
    - After loading message history, also load existing insights via `insightRepository.findByInterviewId(interviewId)`
    - Build context: system prompt + messages + an insight context message (see below)
    - Call `llmProvider.generateResponse()` (returns `{ content: string }` — the JSON)
    - Parse response via `parseInterviewerResponse(response.content)`
    - Persist the assistant message using just the parsed `text` (not the raw JSON)
    - Persist new insights via `insightRepository.createMany()` (filter out any with empty content)
    - Return the response text as a plain string (callers are unaffected)
  - **Insight context injection:**
    - If the interview has existing insights, include them as an additional message in the context:
      ```
      [Previous interview notes]
      - ENTITY: sister Maria — older, bossy dynamic, user showed warmth
      - DETAIL: Teresa mentioned alongside Maria but no details given — worth exploring
      - EMOTION: pride and nostalgia when describing dad's hardware store
      ```
    - Format as the last "user" message before the current exchange, or as a system-level injection — needs testing to see which works better
    - Include on every turn, not just when summarization kicks in. This is what makes insights useful from turn 1.
  - **`startInterview(bookId, questionId)`** changes:
    - Uses the same JSON response format for consistency
    - Parse the opening message via `parseInterviewerResponse()` — insights will typically be empty for the opening (no user content yet)
    - Persist any insights if the AI happens to note something from the question topic
  - **`getInsights(interviewId: string)`** — new method
    - Returns all insights for an interview via `insightRepository.findByInterviewId()`
  - **`getBookInsights(bookId: string)`** — new method
    - Returns all insights across all interviews in a book
    - Loads interviews via `interviewRepository.findByBookId()`, then loads insights for each
    - Enables cross-interview context (wiring into context injection is a future enhancement)

### tRPC Router

- Modify `src/server/routers/interview.ts`:
  - Add `getInsights` procedure — protected query. Input: `{ interviewId: string }`. Returns insights for the interview. Verifies ownership via book → user chain.

## What This Plan Does NOT Do

- **Cross-interview insight injection.** `getBookInsights()` exists as a method, but insights from other interviews are not yet injected into a new interview's context. The infrastructure is ready; the wiring comes later.
- **`explored` tracking.** The `explored` field exists in the data model (Plan 1.1) but this plan doesn't actively set it. The AI sees all its notes and naturally decides what to follow up on. Explicit explored tracking can be added later if the notepad gets too long.
- **Insight deduplication.** The AI is instructed not to repeat insights already in its notes (it can see them in context). If duplicates slip through, they're harmless — a future cleanup pass can deduplicate. No string-similarity logic needed for MVP.
- **Insight UI.** No sidebar or panel showing extracted insights. The `getInsights` tRPC endpoint enables a future UI.

## Notes

- The LLMProvider interface is unchanged — `generateResponse()` still returns `{ content: string }`. The structured output is handled entirely in the service layer via the response parser. This is a key architectural guardrail from ADR 014: switching to Option A means changing the parser and adding a separate extraction call, nothing else.
- The response parser is the critical reliability component. It must never throw — malformed LLM output degrades to "no insights this turn" rather than a crashed conversation.
- The system prompt emphasizes that conversation quality comes first. The AI should never sacrifice warmth or naturalness for the sake of producing better-structured insights.
- Insight content should read like a journalist's notes, not database entries. "Teresa mentioned alongside Maria but user gave no details — worth exploring" is better than "ENTITY: Teresa, role: sister."

## Done When

- The interviewer returns structured JSON responses with both text and insights
- The response parser handles all fallback cases without throwing
- Insights are persisted as Insight records after each assistant response
- Existing insights are injected into context on every turn
- A malformed LLM response degrades gracefully (response text preserved, no insights)
- `trpc.interview.getInsights({ interviewId })` returns extracted insights
- Prompt prototype has been tested and conversation quality validated
- All tests pass via `npm test`
- `npm run build` succeeds
- `npm run lint` passes
