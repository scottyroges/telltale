# Plan 1.6: Context Window Management

**Status:** Planned
**Depends on:** 1.2 (needs conversation service + LLM provider), 1.5 (insights already injected into context every turn — this plan adds summarization)
**Blocks:** None in Phase 1

## Goal

Manage the context window for long conversations — summarize older messages into InterviewSummary records, keep recent ones verbatim. Without this, conversations longer than ~20 messages will hit the context limit or become expensive/slow due to sending the full history every turn.

Note: Plan 1.5 already injects insights into context on every turn. This plan takes over context assembly from the conversation service, adding summarization while preserving the existing insight injection.

**For detailed mental model with examples, see:** `docs/architecture/context-window-management.md`

- **Organize prompts** (do this first):

- Create `src/prompts/summarization.ts`:
  - Exports `SUMMARIZATION_PROMPT` as a string constant
  - Instructs the LLM to summarize a conversation excerpt
  - Instructions: preserve names, dates, places, emotional moments, key narrative details. Be concise but don't lose specificity. Output a prose summary, not bullet points.
- Create `src/services/context.service.ts`:
  - Imports `llmProvider` from `@/lib/llm` (typed as `LLMProvider`)
  - **`buildContextWindow(interviewId: string)`**
    - Loads the interview via `interviewRepository.findById()` to get `bookId`
    - Loads all messages for the interview via `messageRepository.findByInterviewId()`
    - Filters messages to only include USER and ASSISTANT roles (excludes SYSTEM messages)
    - Loads existing insights via `insightRepository.findByInterviewId(interviewId)` (the `bookId` from the interview record is available for logging if needed) (all insights — TODO: filter by `explored = false` when we implement explored tracking)
    - Loads the latest InterviewSummary via `interviewSummaryRepository.findLatestByInterviewId()`
    - **Error handling:** If `interviewRepository.findById()` returns null, throw `Error('Interview not found: ${interviewId}')`. If any other repository call fails, throw the error (caller is responsible for handling)
    - Estimates total token count for **everything going to the LLM** (system prompt + summary + messages + insights) using a heuristic (chars / 4)
    - **Logs token breakdown** using `console.log`: `{ systemPrompt: X tokens, summary: Y tokens, messages: Z tokens, insights: W tokens, total: N tokens }` — visibility into whether any component grows too large
    - If total tokens are under the threshold (8,000 tokens), return all messages with insights injected — no summarization or splitting needed (all messages are considered "recent")
    - If total tokens exceed the threshold:
      - Split messages into three buckets:
        - **Recent** = last `RECENT_WINDOW_SIZE` messages (5 messages, kept verbatim)
        - **Already summarized** = messages covered by existing summary (count tracked in InterviewSummary.messageCount)
        - **Old** = messages between already_summarized and recent (accumulation bucket waiting to be summarized)
      - Calculate split:
        - `recentMessages = allMessages.slice(-RECENT_WINDOW_SIZE)` (last 5)
        - `alreadySummarizedCount = existingSummary?.messageCount ?? 0`
        - `oldMessages = allMessages.slice(alreadySummarizedCount, -RECENT_WINDOW_SIZE)` (everything between summarized and recent)
      - Check if we should create a new summary:
        - If `oldMessages.length >= SUMMARIZATION_BATCH_SIZE` (5 messages): call `summarizeMessages()` to create a new summary
        - This means messages accumulate in the old bucket until we have a batch worth summarizing
      - If `summarizeMessages()` succeeds: create a new InterviewSummary via `interviewSummaryRepository.create()` with:
        - `parentSummaryId` pointing to the previous summary (linked list pattern from 1.1)
        - `messageCount = alreadySummarizedCount + oldMessages.length` (total messages now covered by summary)
      - If summarization fails:
        - Do NOT create an InterviewSummary record (so the next turn will retry when the condition is met again)
        - Fall back to truncation: keep last `SUMMARIZATION_BATCH_SIZE` messages from old region + `RECENT_WINDOW_SIZE` recent messages = 10 total messages (5 + 5)
        - The conversation continues without interruption
    - Assemble and return the context window:
      1. System prompt (from `prompts/interviewer.ts`) — currently returned unchanged; extensible for future modifications (e.g., book context, preferences)
      2. If summarized: a summary message with `{ role: 'assistant', content: 'To recap our earlier conversation: [summary text]' }` — positions the summary as the assistant providing context
      3. Recent messages verbatim (last `RECENT_WINDOW_SIZE` messages) **with insights injected before the last USER message** as `{ role: 'assistant', content: '[insight text]' }` (preserves ADR 018 placement pattern; changes role from 'user' to 'assistant' for semantic accuracy)
    - Returns `{ systemPrompt: string, messages: LLMMessage[] }` where insights are already embedded — ready to pass to `llmProvider.generateResponse()`
    - **Edge cases:** Handles zero messages gracefully (returns empty array for defensive handling). In practice, `startInterview` persists the topic message before calling `buildContextWindow`, so it will have 1 message; `sendMessage` will have 2+ messages.
  - **`summarizeMessages(messages: Message[], existingSummary?: string)`**
    - Takes `Message[]` array and internally formats them as a transcript string for optimal summarization (caller doesn't need to know about formatting)
    - Builds the input: if there's an existing summary, includes it as context ("Previous summary: ...") followed by the formatted transcript of new messages to incorporate
    - Calls `llmProvider.generateResponse()` with the summarization prompt and formatted content (single attempt, no retry logic — fail fast and fall back to truncation)
    - Returns the summary string or throws if LLM call fails
  - **Token counting heuristic:**
    - `estimateTokens(text: string): number` — `Math.ceil(text.length / 4)`
    - Simple but effective for English text. No need for a tokenizer library. We accept the ~10-20% estimation error given the conservative thresholds.
    - Constants:
      - `MAX_CONTEXT_TOKENS = 16000`
      - `SUMMARIZATION_THRESHOLD = 8000` (no safety margin needed — threshold is already conservative)
      - `RECENT_WINDOW_SIZE = 5` (keep last 5 messages verbatim)
      - `SUMMARIZATION_BATCH_SIZE = 5` (summarize when 5 messages accumulate in the old bucket)
- Update `src/services/conversation.service.ts`:
  - Both `startInterview` and `sendMessage` now use `contextService.buildContextWindow(interviewId)` for consistency (enables future book-level insight injection)

## Notes

- **Prompt organization:** All prompts now live in `src/prompts/` for easy review and iteration. This makes it simple to see all LLM instructions in one place.
- The context window strategy is: **three-bucket sliding window** — messages flow from recent → old → already_summarized as the conversation grows. This keeps recent messages verbatim while compressing older messages.
- **Recent window size = 5 messages:** Optimized for long-form storytelling where user messages contain rich detail. The last 5 messages stay verbatim for maximum fidelity.
- **Summarization batch size = 5 messages:** When 5 messages accumulate in the old bucket, we summarize them. This means summarization happens every 5 messages after the initial window fills.
- Summarization is **incremental** — we create new InterviewSummary records in a linked list (each pointing to its parent). Each summary incorporates the previous summary plus new messages. This avoids re-summarizing the entire conversation every turn.
  - **Trade-off:** Incremental summarization (each summary builds on the previous) trades accuracy for efficiency. If early summaries lose details, those details won't be recovered in later summaries. This is an acceptable trade-off for context window management — the logging from token breakdown will help detect if summaries grow too large.
- **Summarization is triggered only when the old bucket fills up** — specifically when `oldMessages.length >= SUMMARIZATION_BATCH_SIZE` (5 messages). Messages accumulate in the old bucket as they age out of the recent window, and we summarize in batches rather than on every turn.
- The InterviewSummary model from 1.1 tracks `messageCount` which represents the total number of USER/ASSISTANT messages covered by that summary (includes all previously summarized messages plus the new batch). This allows the context service to know which messages are already summarized vs. still in the old/recent buckets.
- The threshold constants (8K tokens for summarization trigger, 16K max context) are tuned conservatively for cost/latency management. We accept the ~10-20% token estimation error — the token breakdown logging will surface if this becomes a problem.
- **Insight injection** is already happening every turn from Plan 1.5. This plan moves that logic into `buildContextWindow` for centralized context assembly. All insights are included (TODO: filter by `explored = false` when we implement explored tracking).
  - **Insight extraction still uses `parseWithRetry`** in `conversation.service.sendMessage()` for extracting new insights from LLM responses. Only the injection logic moves to `buildContextWindow`.
  - **Insight role change:** Insights are injected with `role: 'assistant'` (was `role: 'user'` in current code). This is more semantically accurate — insights are the AI's own notes, not user speech. ADR 018 will be updated after implementation to reflect this change.
- The summarization prompt tells the LLM to preserve specificity — "Maria, the sister who lived in Calabria" not "a family member". This is critical because the summary replaces the original messages.
- `buildContextWindow` returns `{ systemPrompt, messages: LLMMessage[] }` where insights are already embedded — provider-agnostic format ready for `llmProvider.generateResponse()`.
- **Fallback behavior:** If summarization fails (LLM error), keep the last `SUMMARIZATION_BATCH_SIZE` messages from old region verbatim instead of summarizing (total 10 messages: 5 old + 5 recent). Single attempt for summarization — fail fast and fall back to truncation. No InterviewSummary record is created on failure, so subsequent turns will retry when the condition is met again (conversation continues, and if the failure was temporary, summarization will eventually succeed).
- **Message format details:**
  - Summary messages: `{ role: 'assistant', content: 'To recap our earlier conversation: [summary text]' }`
  - Insight messages: `{ role: 'assistant', content: '[insight text]' }` injected before the last USER message
  - Both use 'assistant' role to position them as the AI providing context, not the user speaking.

- Conversations longer than the threshold trigger summarization of older messages
- A new InterviewSummary record is created with the correct parentSummaryId (linked list)
- Subsequent turns reuse the stored summary and only incorporate new messages
- All insights are injected into the context window (before the last message)
- The AI references earlier details despite summarization (manual verification)
- A 20+ message conversation works without degradation
