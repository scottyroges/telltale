# Plan 2b.2: Plan Generation Service + Prompt

**Status:** Not Started
**Created:** 2026-03-02
**Parent plan:** 2b (onboarding + interview planning)
**Depends on:** 2b.1 (schema, domain types, repositories)
**Blocks:** 2b.3 (onboarding service + tRPC router)

## Goal

Add the LLM prompt and service that turns a user's life area responses into a personalized interview plan. The service takes a bookId, loads the life area responses, current core memory, and question catalog, calls the LLM, then persists the generated topics and bootstrapped core memory. No UI, no router — testable entirely via unit tests.

After this PR, calling `planGenerationService.generatePlan(bookId)` produces 8-15 personalized InterviewTopic records and a bootstrapped core memory block on the Book.

## Changes

### 1. Plan Generation Prompt

**`src/prompts/plan-generation.ts`** — new file, exports `getPlanGenerationPrompt(params)`.

Function signature:

```typescript
type PlanGenerationPromptParams = {
  lifeAreaResponses: { key: string; name: string; content: string }[];
  skippedAreas: { key: string; name: string }[];
  coreMemory: string | null;
  questionsByLifeArea: { lifeAreaKey: string; questions: { title: string; prompt: string }[] }[];
};

export function getPlanGenerationPrompt(params: PlanGenerationPromptParams): string
```

The function builds a system prompt string (same pattern as `getConversationSystemPrompt` and `getMemorySystemPrompt`). The prompt instructs the LLM to:

- Review life area responses to personalize topics
- Check the question catalog for life dimensions the responses didn't cover (catalog as coverage fallback)
- Respect stated priorities — areas with longer responses matter more to the subject
- Treat skipped areas as signal (not relevant or sensitive — don't force them, but a gentle topic might be appropriate)
- Order topics: broad/foundational first (childhood, family), focused/specific later, forward-looking/thematic at the end (legacy, wisdom)
- Generate 8-15 topics
- Assign a `primaryLifeAreaKey` for display grouping and `lifeAreaKeys` array for all related areas
- Produce a Book Memory section (key people, life narrative, emotional patterns) that incorporates the existing core memory (if any) plus new insights from the life area responses — no Interview Memory section since there's no active interview
- Keep Book Memory at 800-1,500 characters, matching the format from `getMemorySystemPrompt()`

The prompt's output format (described in the prompt itself):

```json
{
  "topics": [
    {
      "title": "Growing up on the farm with three brothers",
      "description": "Explore early family dynamics, daily life, and what shaped your relationship with your siblings.",
      "primaryLifeAreaKey": "childhood",
      "lifeAreaKeys": ["childhood", "relationships"],
      "rationale": "Long response about childhood suggests this is central to their story."
    }
  ],
  "coreMemory": "## Book Memory\nKey people: ..."
}
```

The prompt ends with the standard JSON-only instruction: `Always respond with ONLY valid JSON — no preamble, no markdown fences, no explanation outside the JSON.`

The user message is assembled separately (not part of the system prompt) and contains the actual data:

```
[Life area responses]
childhood: "I grew up on a farm..."
career: "I worked in advertising..."

[Skipped areas]
parenthood, passions

[Current core memory]
## Book Memory
Key people: ...
(or: "No existing memory for this subject.")

[Question catalog by life area]
childhood:
  - "Childhood memories" — Tell me about your earliest memories...
  - ...
career:
  - "First job" — What was your first real job?
  - ...
```

### 2. Plan Generation Response Parser

In `src/services/plan-generation.service.ts`, implement the defensive JSON parser following the same pattern as `parseMemoryResponse` in `memory.service.ts`:

```typescript
function extractCandidate(raw: string): string {
  // Same implementation as memory.service.ts:
  // 1. Try markdown code fence
  // 2. Try bare JSON object
  // 3. Fallback: return as-is
}

type ParsedPlanResponse = {
  topics: {
    title: string;
    description: string;
    primaryLifeAreaKey: string;
    lifeAreaKeys: string[];
    rationale: string;
  }[];
  coreMemory: string | null;
};

function parsePlanResponse(raw: string): ParsedPlanResponse | null
```

Validation rules inside `parsePlanResponse`:
- Extract candidate JSON (fence → bare → raw)
- `JSON.parse` with try-catch → return `null` on failure
- Validate `topics` is a non-empty array
- Validate each topic has `title` (string), `description` (string), `primaryLifeAreaKey` (string), `lifeAreaKeys` (string array)
- Filter out topics with invalid `primaryLifeAreaKey` (not in `LIFE_AREA_KEYS`)
- Validate `coreMemory` is a non-empty string, or null
- Return `null` if no valid topics remain after filtering

Note: `extractCandidate` is the same logic as in `memory.service.ts`. Consider extracting it to a shared `src/lib/json-parse.ts` utility if it becomes a three-time pattern, but for now duplicating in-file is fine (two call sites doesn't warrant abstraction).

### 3. Plan Generation Service

**`src/services/plan-generation.service.ts`** — new file:

```typescript
import { llmProvider } from "@/lib/llm";
import { lifeAreaResponseRepository } from "@/repositories/life-area-response.repository";
import { bookRepository } from "@/repositories/book.repository";
import { questionRepository } from "@/repositories/question.repository";
import { interviewTopicRepository } from "@/repositories/interview-topic.repository";
import { getPlanGenerationPrompt } from "@/prompts/plan-generation";
import { LIFE_AREAS } from "@/domain/life-area";
import { createId } from "@/lib/id";

export const planGenerationService = {
  async generatePlan(bookId: string): Promise<InterviewTopic[]> {
    // 1. Load data
    // 2. Build prompt params
    // 3. Call LLM (non-streaming)
    // 4. Parse response
    // 5. Create InterviewTopic records
    // 6. Update core memory
    // 7. Set onboarding status to COMPLETE
    // 8. Return created topics
  },
};
```

Step-by-step:

**Step 1 — Load data:**
```typescript
const [responses, book, allQuestions] = await Promise.all([
  lifeAreaResponseRepository.findByBookId(bookId),
  bookRepository.findById(bookId),
  questionRepository.findAll(),
]);

if (!book) throw new Error(`Book not found: ${bookId}`);
```

**Step 2 — Build prompt params:**
- Map `responses` to `{ key, name, content }` using `LIFE_AREAS` for the name lookup
- Compute `skippedAreas` as the life areas not present in responses
- Group `allQuestions` by `lifeAreaKey` into `questionsByLifeArea`
- Call `getPlanGenerationPrompt(params)` to get the system prompt
- Build the user message string from the data (life area responses, skipped areas, core memory, question catalog)

**Step 3 — Call LLM:**
```typescript
const response = await llmProvider.generateResponse(
  systemPrompt,
  [{ role: "user", content: userMessage }],
  { maxTokens: 4096 },
);
```

Use `maxTokens: 4096` since the response includes both 8-15 topic objects and a core memory block — more output than the memory service's single JSON object.

**Step 4 — Parse response:**
```typescript
const parsed = parsePlanResponse(response.content);
if (!parsed) {
  throw new Error("[PlanGeneration] Failed to parse LLM response");
}
```

Unlike the memory service (which returns safe defaults on parse failure), plan generation throws — this is a user-initiated action with a loading state, so the caller (Phase 3's onboarding service) can catch and surface the error. Returning silently with no topics would be confusing.

**Step 5 — Create topics:**
```typescript
const topicRecords = parsed.topics.map((t, index) => ({
  id: createId(),
  bookId,
  title: t.title,
  description: t.description,
  primaryLifeAreaKey: t.primaryLifeAreaKey,
  lifeAreaKeys: t.lifeAreaKeys,
  orderIndex: index,
  status: "SUGGESTED" as const,
}));

const topics = await interviewTopicRepository.createMany(topicRecords);
```

**Step 6 — Update core memory:**
```typescript
if (parsed.coreMemory) {
  await bookRepository.updateCoreMemory(bookId, parsed.coreMemory);
}
```

Replaces existing core memory unconditionally — the LLM was given the existing memory as input and told to incorporate it.

**Step 7 — Set onboarding status:**
```typescript
await bookRepository.updateOnboardingStatus(bookId, "COMPLETE");
```

**Step 8 — Return:**
```typescript
return topics;
```

Test cases:

- **Happy path:** Mock LLM returns valid JSON with topics and coreMemory → verify `createMany` called with correct topic records (title, description, primaryLifeAreaKey, lifeAreaKeys, orderIndex, status: "SUGGESTED"), `updateCoreMemory` called with the LLM's coreMemory output, `updateOnboardingStatus` called with "COMPLETE", returns created topics
- **Prompt assembly:** Verify `generateResponse` is called with a system prompt containing key instructions and a user message containing the life area responses, skipped areas, core memory, and questions
- **Core memory is null:** When book has no existing core memory, the user message contains "No existing memory" — verify via `mockGenerateResponse.mock.calls[0]`
- **LLM returns no coreMemory field:** `updateCoreMemory` is NOT called, but topics are still created and status is still set to COMPLETE

- **LLM returns topics with invalid primaryLifeAreaKey:** Those topics are filtered out; remaining valid topics are still created
- **LLM returns empty topics array:** Service throws (no valid topics)
- **Book not found:** Service throws `Book not found: {bookId}`
- **maxTokens:** Verify `generateResponse` is called with `{ maxTokens: 4096 }`
