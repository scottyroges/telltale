# Plan 1.4: Chat UI

**Status:** Planned
**Depends on:** 1.3 (needs tRPC routers)
**Blocks:** 1.5, 1.6

## Goal

Build the minimal chat interface — message display, text input, and response rendering. Completes the first end-to-end loop: a logged-in user can browse questions, start an interview, have a multi-turn conversation, and see the full transcript persist across page refreshes.

## Manual Steps (Before Code)

None.

## Key Steps

- Create `src/app/(app)/interviews/page.tsx`:
  - Server Component — shows the user's book with its questions and interviews
  - Calls `serverTRPC().book.getOrCreate()` to ensure the user has a book
  - Lists BookQuestions with their status (NOT_STARTED, STARTED, COMPLETE)
  - For questions with an active interview, links to `/interview/[interviewId]`
  - For NOT_STARTED questions, shows a "Start Interview" button
  - "Start Interview" calls `trpc.interview.start({ bookId, questionId })`, then navigates to `/interview/[interviewId]`
  - Section to browse and add questions from the global catalog: calls `trpc.question.list` and `trpc.book.addQuestion`
  - Functional styling only (CSS Modules) — not polished
- Create interview page and components:
  - `src/app/(app)/interview/[interviewId]/page.tsx`:
    - Server Component that loads initial messages via `serverTRPC().interview.getMessages({ interviewId })`
    - Passes messages as props to the client-side Chat component
    - Verifies the interview exists and belongs to the user (404 if not)
  - `src/components/interview/chat.tsx`:
    - Client Component — the main chat orchestrator
    - Receives initial messages as props
    - Uses `trpc.interview.sendMessage` mutation for sending messages
    - Manages local message list: initial messages + optimistic user messages + response
    - Shows a loading/thinking indicator while waiting for the AI response
    - After response arrives, adds it to the message list and invalidates the tRPC query to sync with server state
  - `src/components/interview/message-list.tsx`:
    - Renders a scrollable list of messages
    - Auto-scrolls to bottom on new messages
    - Shows a "thinking" indicator while awaiting the AI response
  - `src/components/interview/message-bubble.tsx`:
    - Renders a single message with role-based styling (user messages right-aligned, assistant messages left-aligned)
    - Shows timestamp
  - `src/components/interview/message-input.tsx`:
    - Text input with send button
    - Submit on Enter (Shift+Enter for newline)
    - Disabled while awaiting response
    - Auto-focus on mount
- Update `src/app/(app)/dashboard/page.tsx`:
  - Add a link/button to navigate to `/interviews`
- CSS Modules for all new components:
  - `src/components/interview/chat.module.css`
  - `src/components/interview/message-list.module.css`
  - `src/components/interview/message-bubble.module.css`
  - `src/components/interview/message-input.module.css`
  - `src/app/(app)/interviews/interviews.module.css`
  - Functional layout — nothing fancy, just readable and usable

## Notes

- The interviews page handles the "new interview" flow: browse questions → add to book → start interview → redirect to interview page. This is the simplest possible UX for MVP.
- The interview page is a Server Component that loads initial data, with a Client Component (`Chat`) handling the interactive parts. This follows the Next.js pattern of pushing client boundaries down.
- No custom streaming hook needed — the Chat component uses a standard tRPC mutation (`trpc.interview.sendMessage`). React Query handles loading/error states automatically. Per ADR 013, short interviewer responses don't benefit from token-by-token streaming.
- After a response arrives, the Chat component should invalidate/refetch the `interview.getMessages` query so the server-persisted messages are the source of truth. The optimistic messages are replaced by the real ones.
- Auto-scroll behavior: scroll to bottom when new messages arrive, but only if the user is already at/near the bottom (don't force-scroll if they've scrolled up to read earlier messages).
- The interview page uses a dynamic route `[interviewId]` — each interview is its own conversation started from a specific question.
- CSS Modules keep styles scoped and avoid conflicts. No global styles or CSS-in-JS.
- Interviews page uses a Client Component wrapper for the "Start Interview" and "Add Question" forms that call tRPC mutations. The page itself is a Server Component.

8. See the AI's response appear after a brief loading indicator
9. AI asks a meaningful follow-up question (not yes/no)
10. Have a multi-turn conversation (3+ exchanges)
11. Refresh the page — full transcript is preserved
12. `npm test` passes
13. `npm run build` succeeds
