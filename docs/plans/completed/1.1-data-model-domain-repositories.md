## Data Model

### Entity Relationships

```
Question (global prompt catalog)

User
└── Book (grouping container)
    ├── BookQuestion (join: which questions are in this book)
    ├── Interview (bookId, questionId — the raw conversation)
    │   ├── Message (the back-and-forth)
    │   ├── Insight (extracted metadata for follow-ups)
    │   └── InterviewSummary (rolling summaries, linked list)
    └── Story (bookId, interviewId — polished output)
        └── StorySection (ordered chunks of generated prose)
```

### Question

Global prompt catalog — not user-specific, shared across all users and books. Questions are the starting points that kick off interviews.

- `id` (cuid), `category` (String), `prompt` (String), `orderIndex` (Int)
- `createdAt`, `updatedAt`
- No user relation — these are global seed data

### Book

Top-level grouping container. A user can have multiple books, each containing a set of interviews and resulting stories.

- `id` (cuid), `userId`, `title`, `status` (BookStatus)
- `createdAt`, `updatedAt`
- Status lifecycle: `IN_PROGRESS` → `COMPLETE` → `ARCHIVED`

### BookQuestion

Join table tracking which questions a user has selected for a given book. Provides the "checklist" of interview prompts for the book.

- `id` (cuid), `bookId`, `questionId`, `orderIndex` (Int), `status` (BookQuestionStatus)
- `createdAt`, `updatedAt`
- Status lifecycle: `NOT_STARTED` → `STARTED` → `COMPLETE`
- Unique constraint on `(bookId, questionId)` — a question can only appear once per book

### Interview

The raw conversation container. An interview is started from a question within a book. Contains all messages, extracted insights, and rolling summaries.

- `id` (cuid), `bookId`, `questionId`, `status` (InterviewStatus)
- `createdAt`, `updatedAt`
- Status lifecycle: `ACTIVE` → `PAUSED` → `COMPLETE`
- No topic field — the originating Question provides the starting point; summaries and insights capture where the conversation actually went

### Message

Individual message in an interview conversation. Messages are append-only and immutable.

- `id` (cuid), `interviewId`, `role` (MessageRole), `content` (String)
- `createdAt` (no `updatedAt` — messages are immutable)
- Roles: `USER`, `ASSISTANT`, `SYSTEM`

### Insight

Extracted metadata from interview conversations. Used to power follow-up questions within an interview and across interviews in a book. Replaces the original "StoryThread" concept.

- `id` (cuid), `bookId`, `interviewId`, `type` (InsightType), `content` (String), `explored` (Boolean, default false)
- `createdAt`, `updatedAt`
- Types: `ENTITY` (person, place, thing), `EVENT` (something that happened), `EMOTION` (emotional moment worth revisiting), `DETAIL` (unexplored detail mentioned but not elaborated on)
- Interview-scoped but queried across all interviews in a book for cross-interview follow-ups

### InterviewSummary

Rolling summaries for context window management. Forms a linked list — each summary incorporates the previous summary plus new messages. When the AI needs context, it uses the latest summary + messages after it.

- `id` (cuid), `interviewId`, `parentSummaryId` (self-ref, nullable — null for first summary), `content` (String), `messageCount` (Int — cumulative messages covered)
- `createdAt`

### Story

Polished narrative output generated from an interview. An interview can produce one or more stories. Stories have a direct relationship to Book for independent ordering.

- `id` (cuid), `bookId`, `interviewId`, `title` (String), `prose` (String, nullable — assembled from sections), `orderIndex` (Int), `status` (StoryStatus)
- `createdAt`, `updatedAt`
- Status lifecycle: `DRAFT` → `REVIEWED` → `FINAL`

### StorySection

Ordered chunks of generated prose. LLMs can't reliably produce long-form stories in one pass, so stories are generated section by section and assembled. Provides retry granularity — if one section is bad, regenerate just that one.

- `id` (cuid), `storyId`, `orderIndex` (Int), `content` (String), `status` (StorySectionStatus)
- `createdAt`, `updatedAt`
- Status lifecycle: `GENERATING` → `DRAFT` → `FINAL`

## Domain Types

Domain types are what the application works with — they are decoupled from Prisma types and defined as plain TypeScript types with no runtime code. They can match the DB schema exactly or combine data from multiple tables. Repositories are responsible for mapping Prisma results to domain types.

### Question

```typescript
export type Question = {
  id: string;
  category: string;
  prompt: string;
  orderIndex: number;
  createdAt: Date;
  updatedAt: Date;
};
```

### Book

```typescript
export type BookStatus = "IN_PROGRESS" | "COMPLETE" | "ARCHIVED";

export type Book = {
  id: string;
  userId: string;
  title: string;
  status: BookStatus;
  createdAt: Date;
  updatedAt: Date;
};
```

### BookWithDetails

Composite type returned by `bookRepository.findByIdWithDetails()` — includes BookQuestions (with their Question data) and Interviews.

```typescript
export type BookWithDetails = Book & {
  bookQuestions: (BookQuestion & { question: Question })[];
  interviews: Interview[];
};
```

### BookQuestion

```typescript

export type BookQuestion = {
  id: string;
  bookId: string;
  questionId: string;
  orderIndex: number;
  status: BookQuestionStatus;
  createdAt: Date;
  updatedAt: Date;
};
```

### Interview

```typescript
export type InterviewStatus = "ACTIVE" | "PAUSED" | "COMPLETE";

export type Interview = {
  id: string;
  bookId: string;
  questionId: string;
  status: InterviewStatus;
  createdAt: Date;
  updatedAt: Date;
};
```

### Message

```typescript
export type MessageRole = "USER" | "ASSISTANT" | "SYSTEM";

export type Message = {
  id: string;
  interviewId: string;
  role: MessageRole;
  content: string;
  createdAt: Date;
};
```

No `updatedAt` — messages are immutable.

### Insight

The interviewer's "mental notes" — produced inline as part of each response (ADR 014, Option B). Insights capture what the AI noticed, what it wants to explore later, and emotional nuance. They are persisted per turn and injected back into context on every subsequent turn.

```typescript
export type InsightType = "ENTITY" | "EVENT" | "EMOTION" | "DETAIL";

export type Insight = {
  id: string;
  bookId: string;
  interviewId: string;
  type: InsightType;
  content: string;
  explored: boolean;
  createdAt: Date;
  updatedAt: Date;
};
```

- `ENTITY` — person, place, thing (e.g., "sister Maria", "farm in Calabria")
- `EVENT` — something that happened
- `EMOTION` — emotional moment worth revisiting
- `DETAIL` — unexplored detail mentioned but not elaborated on
- `explored` — tracks whether the insight has been followed up on (deferred to a future plan, defaults to `false`)
- Content should read like a journalist's notes: "Teresa mentioned alongside Maria but no details given — worth exploring" not "ENTITY: Teresa, role: sister"

### InterviewSummary

```typescript
export type InterviewSummary = {

  interviewId: string;

  messageCount: number;
  createdAt: Date;

```

No `updatedAt` — summaries are append-only (new summaries are created, never edited).

## Key Steps

- Update `prisma/schema.prisma` with new models and enums:
  - Enums: `BookStatus`, `BookQuestionStatus`, `InterviewStatus`, `MessageRole`, `InsightType`, `StoryStatus`, `StorySectionStatus`
  - All application models use `@default(cuid())` for IDs
  - All models use `@@map("snake_case")` table names, `@@index` on foreign keys, `onDelete: Cascade` on parent relations
  - Add `books Book[]` relation to User model
  - `BookQuestion` gets `@@unique([bookId, questionId])`
  - `InterviewSummary` gets self-referential `parentSummaryId`
- Create domain types as separate files in `src/domain/`:
  - `src/domain/question.ts` — `Question` type
  - `src/domain/book.ts` — `Book` type, `BookStatus` union type
  - `src/domain/book-question.ts` — `BookQuestion` type, `BookQuestionStatus` union type
  - `src/domain/interview.ts` — `Interview` type, `InterviewStatus` union type
  - `src/domain/message.ts` — `Message` type, `MessageRole` union type
  - `src/domain/insight.ts` — `Insight` type, `InsightType` union type
  - `src/domain/interview-summary.ts` — `InterviewSummary` type
  - `src/domain/story.ts` — `Story` type, `StoryStatus` union type
  - `src/domain/story-section.ts` — `StorySection` type, `StorySectionStatus` union type
  - `src/domain/types.ts` — re-exports all domain types
  - Domain types are plain TypeScript types/interfaces — no runtime code, no Prisma imports
- Create repositories following the `user.repository.ts` pattern:
  - `src/repositories/question.repository.ts`:
    - `create(data: { category, prompt, orderIndex })` — creates a Question
    - `findById(id)` — returns `Question | null`
    - `findAll()` — returns all questions, ordered by orderIndex
    - `findByCategory(category)` — returns questions filtered by category
  - `src/repositories/book.repository.ts`:
    - `create(data: { userId, title })` — creates a Book with IN_PROGRESS status
    - `findById(id)` — returns `Book | null`
    - `findByIdWithDetails(id)` — returns `BookWithDetails | null` (Book + BookQuestions with Question data + Interviews)
    - `findByUserId(userId)` — returns `Book[]` for a user (bare book records, no relations)
    - `updateStatus(id, status)` — updates book status
  - `src/repositories/book-question.repository.ts`:
    - `create(data: { bookId, questionId, orderIndex })` — creates a BookQuestion
    - `findByBookId(bookId)` — returns all BookQuestions for a book, ordered by orderIndex
    - `updateStatus(id, status)` — updates BookQuestion status
    - `delete(id)` — deletes a BookQuestion by id
  - `src/repositories/interview.repository.ts`:
    - `create(data: { bookId, questionId })` — creates an Interview with ACTIVE status
    - `findById(id)` — returns Interview or null
    - `findByBookId(bookId)` — returns all interviews for a book
    - `updateStatus(id, status)` — updates interview status
  - `src/repositories/message.repository.ts`:
    - `create(data: { interviewId, role, content })` — creates a Message
    - `findByInterviewId(interviewId)` — returns all messages for an interview, ordered by createdAt
  - `src/repositories/insight.repository.ts`:
    - `createMany(insights: Array<{ bookId, interviewId, type, content }>)` — bulk creates insights, returns count (callers load fresh via `findByInterviewId` on the next turn)
    - `findByInterviewId(interviewId)` — returns `Insight[]` for an interview
    - `findByBookId(bookId)` — returns `Insight[]` across all interviews in a book
    - `markExplored(id)` — sets explored = true
  - `src/repositories/interview-summary.repository.ts`:
    - `create(data: { interviewId, parentSummaryId?, content, messageCount })` — creates a summary
    - `findLatestByInterviewId(interviewId)` — returns the most recent summary for an interview
  - `src/repositories/story.repository.ts`:
    - `create(data: { bookId, interviewId, title, orderIndex })` — creates a Story with DRAFT status
    - `findById(id)` — returns Story or null
    - `findByBookId(bookId)` — returns all stories for a book, ordered by orderIndex
    - `findByInterviewId(interviewId)` — returns all stories for an interview
    - `updateProse(id, prose)` — sets assembled prose
    - `updateStatus(id, status)` — updates story status
  - `src/repositories/story-section.repository.ts`:
    - `create(data: { storyId, orderIndex, content })` — creates a StorySection with GENERATING status
    - `findByStoryId(storyId)` — returns all sections for a story, ordered by orderIndex
    - `updateContent(id, content)` — updates section content
    - `updateStatus(id, status)` — updates section status
- Write unit tests for each repository in `tests/repositories/`:
  - Follow the health router test pattern: mock `server-only`, mock `@/lib/prisma` with method-level mocks
  - Use `@vitest-environment node` directive
  - Test each repository method: correct Prisma calls, correct arguments, return values

## Notes

- Application models use `@default(cuid())` for IDs — unlike Better Auth models which use plain `String @id` with runtime-generated IDs
- `Message` has no `updatedAt` — messages are append-only, never edited
- `InterviewSummary` has no `updatedAt` — summaries are append-only (new summaries are created, not edited)
- **Repository → Domain mapping pattern:** All repository methods return domain types, not raw Prisma types. Repositories use Prisma's `select` to pick exactly the fields defined in the domain type. For composite types (e.g., `BookWithDetails`), repositories use `include` and map the result to the composite domain type. This keeps services and routers decoupled from Prisma — during backend extraction, only the repository internals change. See `docs/architecture/system-overview.md` for the canonical pattern description.
- Repository tests mock Prisma at the module level (not integration tests against a real DB). Integration tests can be added later with a test database.
- Question table is seeded with global prompts — no user-specific data. Seed script can be added as a follow-up.
- Insights are interview-scoped but will be queried across all interviews in a book for cross-interview follow-up context. Insights are the interviewer's "mental notes" produced inline via Option B (ADR 014) — not a separate extraction call.
- Depth score concept has been deferred — see `docs/ideas/depth-score.md`.
- **Post-completion update (ADR 015):** Repository internals were migrated from Prisma client to Kysely in Plan 1.1.1. Domain types, services, and routers were unaffected — only repository files and their test mocks changed.
