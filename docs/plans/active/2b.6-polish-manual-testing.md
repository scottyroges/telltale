# Plan 2b.6: Polish + Edge Cases + Manual Testing

**Status:** Not Started
**Created:** 2026-03-02
**Parent plan:** 2b (onboarding + interview planning)
**Depends on:** 2b.5 (plan / progress view)

## Goal

Polish the onboarding flow, tune the plan generation prompt against real LLM output, and create a manual test script covering all end-to-end paths. This phase is about quality — verifying that everything from Phases 1-5 works together smoothly and that the generated plans are actually good.

No major new features. The deliverables are UX fixes, prompt refinements, and a manual test plan document.

## Changes

### 1. UX Refinements

These items should mostly be handled by earlier phases. This phase verifies they work and fixes gaps.

**Loading/progress state during plan generation:**

Phase 4 has a "Generating your interview plan..." button text. Verify this is sufficient for the ~10-30 second LLM call. If the delay feels too long without feedback, consider:
- A more descriptive progress message (e.g., "Reading your responses and creating personalized topics...")
- A subtle animation or spinner on the button or below it
- Disabling all textareas during generation to prevent edits while the call is in flight

**"Back to interviews" escape:**

Phase 4's onboarding page has a placeholder `{/* Back link, heading, description */}`. Ensure:
- A visible "Back to interviews" link at the top of the page (matching the `← Books` link pattern from the interviews page)
- The link goes to `/book/{bookId}/interviews`
- The link is available at all times — even during plan generation (the auto-save on blur means no work is lost)

**Resume case verification:**

Phase 4 handles pre-filling via `initialResponses`. Verify the full round-trip:
- Fill 3 areas, navigate away (click back link or browser back)
- Return to `/book/{bookId}/onboarding` — all 3 areas pre-fill with saved content
- The onboarding status is `IN_PROGRESS` and the interviews page shows "Continue setting up..." CTA
- Edit an existing response, blur — the upsert overwrites the old content

### 2. Prompt Tuning

Run the full onboarding flow 3-5 times with varied life area responses (different personas, different areas filled/skipped, different response lengths) and evaluate the LLM output. Adjust `src/prompts/plan-generation.ts` based on findings.

**Topic quality checklist:**
- Are topics specific and personalized (not generic "tell me about your childhood")?
- Do topics pull from the actual details the user shared?
- Are topics ordered sensibly (foundational → specific → forward-looking)?
- Is the count reasonable (8-15 as instructed)?
- Do skipped areas get a gentle topic or are they respected?
- Are `primaryLifeAreaKey` assignments correct — does each topic appear under the right life area in the plan view?
- Do `lifeAreaKeys` arrays capture cross-cutting topics (e.g., a father topic touches childhood + relationships)?

**Core memory quality checklist:**
- Does the bootstrapped Book Memory capture key people, life narrative, and emotional patterns from the responses?
- Is it in the same format as the interviewer's memory prompt (`## Book Memory` section)?
- Does it stay within the 800-1,500 character target?
- When the book already has a core memory (from prior interviews), does the new memory incorporate it without losing information?
- Does interview #1 after onboarding feel informed — does the AI reference things from the life area responses?

**Catalog coverage fallback:**
- Fill only 2-3 life areas (skip the rest)
- Verify the generated topics include some from life areas that were skipped — derived from catalog questions
- The prompt should note which areas were skipped and use catalog questions to suggest gentle topics for those areas

### 3. Threshold Tuning

The `MIN_RESPONSES = 2` threshold in `onboarding.service.ts` may need adjustment after real testing. Consider:
- Is 2 enough to generate a useful plan? (Likely yes — even 2 rich responses give the LLM plenty to work with)
- Would 3 be better? Test with exactly 2 responses and evaluate topic quality
- The generate button already shows the count requirement ("fill at least 2 areas") — adjust the messaging if the threshold changes

### 4. Manual Test Plan

Create `docs/plans/active/2b-manual-test-plan.md` following the same format as `docs/plans/completed/2a-manual-test-plan.md`.

## Manual Test Plan

**Purpose:** Validate the complete onboarding → plan → interview flow against a running app with a real database.

**Prerequisites:**
- Local dev environment running (`npm run dev`)
- Database access (e.g., `psql` or a GUI client connected to the local Postgres)
- A registered, approved user account

### SQL Helpers

```sql
-- Book with onboarding status
SELECT id, title, "onboardingStatus", LEFT("coreMemory", 500) AS memory_preview,
       LENGTH("coreMemory") AS memory_length
FROM book WHERE id = '<bookId>';

-- Full core memory
SELECT "coreMemory" FROM book WHERE id = '<bookId>';

-- Life area responses for a book
SELECT "lifeAreaKey", LEFT(content, 100) AS content_preview, LENGTH(content) AS length
FROM life_area_response WHERE "bookId" = '<bookId>'
ORDER BY "createdAt";

-- Interview topics for a book
SELECT id, title, "primaryLifeAreaKey", status, "orderIndex"
FROM interview_topic WHERE "bookId" = '<bookId>'
ORDER BY "orderIndex";

-- Interviews with topic link
SELECT i.id, i.topic, i.status, i."topicId",
       it.title AS topic_title
FROM interview i
LEFT JOIN interview_topic it ON i."topicId" = it.id
WHERE i."bookId" = '<bookId>'
ORDER BY i."createdAt";

-- Threads for a book (empty until 2c)
SELECT id, title, status, "interviewId", "topicId"
FROM thread WHERE "bookId" = '<bookId>';
```

### Scenario 1: New Book — Full Onboarding Flow

**Goal:** Verify the happy path from book creation through onboarding to first interview.

| Step | Action | Expected |
|------|--------|----------|
| 1 | Create a new book | Book appears on dashboard |
| 2 | Navigate to the book's interviews page | "Get personalized interview topics" CTA is visible. Existing sections (topic input, catalog) also visible below |
| 3 | Query DB: check `onboardingStatus` | `NOT_STARTED` |
| 4 | Click the CTA → onboarding page loads | All 10 life areas visible with name, prompt, and empty textarea |
| 5 | Fill 4-5 life areas with real, varied content (different lengths) | Textareas accept input normally |
| 6 | Click away / blur each textarea after filling | No visible save indicator needed — auto-save fires silently |
| 7 | Query DB: check life area responses | Responses exist for each filled area. `onboardingStatus` is `IN_PROGRESS` |
| 8 | Click "Generate My Interview Plan" | Button shows loading state ("Generating..."), textareas optionally disabled |
| 9 | Wait for LLM response (~10-30 seconds) | Redirected to interviews page |
| 10 | Query DB: check `onboardingStatus` | `COMPLETE` |
| 11 | Query DB: check interview topics | 8-15 topics with `SUGGESTED` status, sensible `primaryLifeAreaKey` assignments |
| 12 | Query DB: check `coreMemory` | Populated with `## Book Memory` section. Contains key people/narrative from responses. 800-1,500 chars |
| 13 | Verify plan view on interviews page | Topics grouped under life area headings. Progress indicators show "0 of N completed". Thread section hidden (empty) |
| 14 | Click "Begin" on a suggested topic | Interview starts, navigated to interview session |
| 15 | Query DB: check topic status | Topic is `ACCEPTED`. Interview has `topicId` set |
| 16 | Send a message in the interview | AI response is informed — references details from life area responses (via bootstrapped core memory) |

**What to look for:**
- Topics are personalized, not generic
- Topics are grouped under the correct life areas
- Core memory captures the person's story accurately
- The first interview feels informed, not blind
