# Plan 1.5: Insight Extraction

**Status:** Planned
**Depends on:** 1.2 (needs conversation service + LLM provider), 1.4 (needs working conversations to test against)
**Blocks:** 1.6

## Goal

After each assistant response, extract structured insights (people, places, events, emotions, unexplored details) via a separate LLM call through the provider interface. Persist as Insight records. Insights power follow-up question intelligence and cross-interview context within a book.

## Manual Steps (Before Code)

None.

## Key Steps

- Create `src/services/extraction-prompt.ts`:
  - Exports `INSIGHT_EXTRACTION_PROMPT` as a string constant
  - Instructs the LLM to analyze the conversation and extract structured insights
  - Requests JSON output in a specific format:
    ```json
    {
      "insights": [
        { "type": "ENTITY", "content": "sister Maria — mentioned growing up together in Calabria" },
        { "type": "EVENT", "content": "the 1978 flood that destroyed the family farm" },
        { "type": "EMOTION", "content": "pride when talking about first job at the bakery" },
        { "type": "DETAIL", "content": "mentioned a 'secret recipe' but didn't elaborate" }
      ]
    }
    ```
  - Instructions: be specific (include names, dates, places), note emotional tone, flag details mentioned but not explored, avoid duplicating insights that have already been extracted
- Create `src/services/insight-extraction.service.ts`:
  - Imports `llmProvider` from `@/lib/llm` (typed as `LLMProvider`)
  - **`extractInsights(interviewId: string, recentMessages: Message[])`**
    - Loads existing insights for the interview via `insightRepository.findByInterviewId()`
    - Builds prompt: extraction system prompt + recent messages + list of already-extracted insights (to avoid duplicates)
    - Calls `llmProvider.generateResponse()` with the extraction prompt and messages
    - Parses JSON response, validates structure
    - Filters out insights that are too similar to existing ones (simple string similarity or exact match on content)
    - Creates new insights via `insightRepository.createMany()`
    - Returns the newly created insights
  - **`getInsights(interviewId: string)`**
    - Returns all insights for an interview via `insightRepository.findByInterviewId()`
  - **`getBookInsights(bookId: string)`**
    - Returns all insights across all interviews in a book — used for cross-interview follow-up context
    - Loads all interviews for the book via `interviewRepository.findByBookId()`, then loads insights for each
- Integrate into `src/services/conversation.service.ts`:
  - After `sendMessage` returns and the assistant message is persisted, fire insight extraction
  - **Fire-and-forget**: insight extraction runs asynchronously and does NOT block the response to the client. The response is already returned when extraction starts.
  - Use a simple `extractInsights(...).catch(console.error)` pattern — if extraction fails, the conversation is unaffected
  - Pass the last N messages (e.g., last 6 messages — 3 exchanges) as context for extraction, not the entire history
- Add `getInsights` to the interview tRPC router:
  - `getInsights` — protected query. Input: `{ interviewId: string }`. Returns insights for the interview. Verifies ownership.
- Write unit tests:
  - `tests/services/insight-extraction.service.test.ts`:
    - Mock `@/lib/llm` (the provider instance)
    - Mock insightRepository
    - Test: extracts insights from messages, deduplicates against existing, creates new records
    - Test: handles malformed JSON gracefully (doesn't throw)
    - Test: handles empty extraction result
    - Test: passes existing insights to prompt for deduplication context

## Notes

- Insight extraction is **non-blocking** — it must never slow down the conversation. The user gets the response immediately, and extraction happens in the background.
- The extraction call uses `llmProvider.generateResponse()` (not streaming) because we need the full structured JSON response.
- Model choice is handled by the provider — the conversation service and extraction service both go through `llmProvider`. If we want to use a smaller/faster model for extraction later, we can create a second provider instance wired to a different model.
- Deduplication is simple for MVP: compare extracted insight content against existing insight content. Exact or near-exact matches are skipped. Sophisticated semantic deduplication can come later.
- The extraction prompt asks the LLM to be specific — "sister Maria" not "a family member", "1978 flood" not "a natural disaster". Specificity makes insights useful for follow-up questions.
- Error handling: if the response isn't valid JSON, log the error and skip extraction for this turn. Don't crash the service.
- The `getInsights` tRPC procedure enables a future UI that shows extracted insights (sidebar or insight panel). The UI for displaying insights is NOT part of this plan.
- Insights are interview-scoped but queryable across a book — `getBookInsights` enables cross-interview context where the AI can reference something mentioned in a different interview.
