# Plan 2b.5: Plan / Progress View

**Status:** Not Started
**Created:** 2026-03-02
**Parent plan:** 2b (onboarding + interview planning)
**Depends on:** 2b.4 (onboarding UI)
**Blocks:** 2b.6 (polish + edge cases)

## Goal

Transform the interviews page into a plan/progress view after onboarding completes. Users see their topics grouped by life area, start interviews from topics, skip topics, and browse open threads. The question catalog and free-text topic input move to a secondary position but remain available. Users can also add catalog questions as interview topics to their plan.

After this PR, a user who completed onboarding sees a structured plan view instead of the flat interviews list, and can start interviews directly from suggested topics.

Thin passthrough. The topic status transition rules from the parent plan (SUGGESTED → ACCEPTED, SUGGESTED → SKIPPED, SKIPPED → SUGGESTED, etc.) are validated at the repository or caller level for now — keep it simple.

**`addTopicFromQuestion(bookId, questionId)`:**

```typescript
async addTopicFromQuestion(
  bookId: string,
  questionId: string,
): Promise<InterviewTopic> {
  const question = await questionRepository.findById(questionId);
  if (!question) throw new Error(`Question not found: ${questionId}`);

  const existingTopics = await interviewTopicRepository.findByBookId(bookId);
  const nextOrderIndex =
    existingTopics.length > 0
      ? Math.max(...existingTopics.map((t) => t.orderIndex)) + 1
      : 0;

  const topics = await interviewTopicRepository.createMany([
    {
      id: createId(),
      bookId,
      title: question.title,
      description: question.prompt,
      primaryLifeAreaKey: question.lifeAreaKey,
      lifeAreaKeys: [question.lifeAreaKey],
      orderIndex: nextOrderIndex,
      status: "ACCEPTED",
    },
  ]);

  return topics[0];
},
```

User-chosen topics start at `ACCEPTED` (they've already been chosen by adding them from the catalog). Uses `question.title` → `topic.title` and `question.prompt` → `topic.description` as specified in the parent plan.

**`src/services/onboarding.service.ts`** — update `getStatus`:

Add `threads` to the return type:

```typescript
async getStatus(bookId: string): Promise<{
  onboardingStatus: OnboardingStatus;
  responses: LifeAreaResponse[];
  topics: InterviewTopic[];
  threads: Thread[];
}> {
  const book = await bookRepository.findById(bookId);
  if (!book) throw new Error(`Book not found: ${bookId}`);

  const [responses, topics, threads] = await Promise.all([
    lifeAreaResponseRepository.findByBookId(bookId),
    interviewTopicRepository.findByBookId(bookId),
    threadRepository.findByBookId(bookId),
  ]);

  return { onboardingStatus: book.onboardingStatus, responses, topics, threads };
},
```

The plan view needs threads alongside topics. Loading them together avoids an extra query.

### 2. Extend Interview Router for topicId

**`src/server/routers/interview.ts`** — add `topicId` to the `start` mutation:

```typescript
start: approvedProcedure
  .input(
    z.object({
      bookId: z.string(),
      topic: z.string().min(5).max(500),
      bookQuestionId: z.string().optional(),
      topicId: z.string().optional(),  // NEW
    }),
  )
  .mutation(async ({ ctx, input }) => {
    await verifyBookOwnership(input.bookId, ctx.userId);

    // Existing bookQuestionId validation...

    if (input.topicId) {
      const topic = await interviewTopicRepository.findById(input.topicId);
      if (!topic || topic.bookId !== input.bookId) {
        throw new TRPCError({
          code: "BAD_REQUEST",
          message: "Topic not found or does not belong to this book",
        });
      }
    }

    const result = await conversationService.startInterview(
      input.bookId,
      input.topic,
      ctx.userName,
    );

    // Existing bookQuestionId linking...

    if (input.topicId) {
      await interviewRepository.setTopicId(result.interviewId, input.topicId);
      await interviewTopicRepository.updateStatus(input.topicId, "ACCEPTED");
    }

    return result;
  }),
```

Mirrors the existing `bookQuestionId` pattern: validate → start interview → link + update status.

**`src/repositories/interview.repository.ts`** — add `topicId` to columns and add `setTopicId` method:

Add `"topicId"` to the `columns` array (Phase 1a added the field to the schema and domain type).

```typescript
async setTopicId(interviewId: string, topicId: string): Promise<Interview> {
  return db
    .updateTable("interview")
    .set({ topicId, updatedAt: new Date() })
    .where("id", "=", interviewId)
    .returning([...columns])
    .executeTakeFirstOrThrow();
},
```

### 3. Plan View Components

**`src/components/book-interviews/plan-view.tsx`** — new client component:

The top-level plan/progress layout. Receives all data as props from the server component:

```typescript
type PlanViewProps = {
  bookId: string;
  topics: InterviewTopic[];
  interviews: Interview[];
  threads: Thread[];
  allQuestions: Question[];
};
```

Behavior:
- Groups topics by `primaryLifeAreaKey` using `LIFE_AREAS` for display names and ordering
- Only shows life areas that have at least one topic
- For each topic, finds associated interviews where `interview.topicId === topic.id`
- Renders a `LifeAreaSection` for each populated life area
- Renders a `ThreadList` below the life area sections (hidden when empty)
- Renders an expandable "Start a new interview" section with the free-text `TopicInput` and `QuestionCatalog`
- The catalog's "Add" action changes for plan users: instead of `book.addQuestion`, it calls `onboarding.addTopicFromQuestion` then refreshes

**`src/components/book-interviews/life-area-section.tsx`** — new client component:

A collapsible section for one life area:

```typescript
type LifeAreaSectionProps = {

```

- Shows life area name as heading

- Collapsible (open by default)
- Renders a `TopicCard` for each topic, ordered by `orderIndex`

**`src/components/book-interviews/topic-card.tsx`** — new client component:

Individual topic with status indicator and actions:

```typescript
type TopicCardProps = {
  bookId: string;
  topic: InterviewTopic;
  interviews: Interview[];
};
```

Display:
- Title and description
- Status indicator: `SUGGESTED` (neutral), `ACCEPTED` (in progress), `COMPLETED` (done), `SKIPPED` (de-emphasized/muted)
- If interviews exist: links to each interview (`/interview/{interviewId}`)

Actions (based on status):
- `SUGGESTED`: "Begin" button (starts interview via `interview.start` with `topicId`) and "Skip" button (calls `onboarding.updateTopicStatus` with `SKIPPED`)
- `ACCEPTED`: "Continue" link to the most recent interview, or "Begin" if no interview yet
- `COMPLETED`: "Review" link to interview
- `SKIPPED`: "Restore" button (sets status back to `SUGGESTED`)

The "Begin" action calls `interview.start` with `{ bookId, topic: topic.title, topicId: topic.id }`, then navigates to `/interview/{interviewId}` on success.

### 4. Interviews Page Changes

**`src/app/(app)/book/[bookId]/interviews/page.tsx`** — conditional rendering:

The page already loads `onboardingStatus` (from Phase 4). When `COMPLETE`, render the plan view instead of the existing layout:

```typescript
export default async function InterviewsPage({ params }: InterviewsPageProps) {
  const { bookId } = await params;
  const trpc = await serverTRPC();
  const [book, allQuestions, { onboardingStatus, topics, threads }] =
    await Promise.all([
      trpc.book.getById({ id: bookId }),
      trpc.question.list(),
      trpc.onboarding.getStatus({ bookId }),
    ]);

  const sortedInterviews = sortInterviews(book.interviews);

  return (
    <div className={styles.page}>
      <Link href="/books" className={styles.back}>&larr; Books</Link>
      <h1 className={styles.heading}>{book.title}</h1>

      {onboardingStatus === "COMPLETE" ? (
        <PlanView
          bookId={bookId}
          topics={topics}
          interviews={sortedInterviews}
          threads={threads}
          allQuestions={allQuestions}
        />
      ) : (
        <>
          {/* OnboardingCta from Phase 4 (NOT_STARTED / IN_PROGRESS) */}
          {/* Existing sections: interviews, topic input, questions, catalog */}
        </>
      )}
    </div>
  );
}
```

The heading and back link are shared across both views. The CTA from Phase 4 only renders in the non-COMPLETE branch.

### 5. Question Catalog — Add Topic Action

When the plan view is active, the catalog's "Add" button should create an `InterviewTopic` instead of a `BookQuestion`. Two approaches:

**Option chosen:** Add an `onAdd` callback prop to `QuestionCatalog` so the parent controls what happens on add. The plan view passes a callback that calls `onboarding.addTopicFromQuestion`, while the non-plan view keeps the existing `book.addQuestion` behavior.

```typescript
type QuestionCatalogProps = {
  bookId: string;
  allQuestions: Question[];
  existingQuestionIds: string[];
  onAdd?: (bookId: string, questionId: string) => void;  // NEW
};
```

When `onAdd` is provided, the component uses it instead of the built-in `book.addQuestion` mutation. The plan view's `PlanView` component provides the `onAdd` that calls `onboarding.addTopicFromQuestion` and then `router.refresh()`.

If the component complexity grows too much with the conditional, extract a thin `PlanCatalog` wrapper instead — but try the callback approach first.
