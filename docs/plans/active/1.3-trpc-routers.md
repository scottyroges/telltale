# Plan 1.3: tRPC Routers

**Status:** Planned
**Depends on:** 1.2 (needs conversation service)
**Blocks:** 1.4

## Goal

Create tRPC routers for book management, question browsing, and interview operations (start, send message, complete). This completes the backend API surface for the conversation engine — all through tRPC with full type safety.

## Manual Steps (Before Code)

None.

## Key Steps

- Create `src/server/routers/question.ts`:
  - `list` — protected query. Returns all questions ordered by orderIndex. Delegates to `questionRepository.findAll()`.
  - `listByCategory` — protected query. Input: `{ category: string }`. Returns questions filtered by category. Delegates to `questionRepository.findByCategory()`.
- Create `src/server/routers/book.ts`:
  - `getOrCreate` — protected mutation. Gets the user's book, creating one if none exists (title defaults to "My Life Story"). For MVP, each user has a single book — this simplifies the UX. Delegates to `bookRepository.findByUserId()` then conditionally `bookRepository.create()`. Returns the book.
  - `getById` — protected query. Input: `{ id: string }`. Returns book with its BookQuestions (including related Question data) and Interviews. Verifies the book belongs to the user.
  - `addQuestion` — protected mutation. Input: `{ bookId: string, questionId: string }`. Adds a question to the book by creating a BookQuestion. Sets orderIndex to the next available position. Verifies the book belongs to the user. Delegates to `bookQuestionRepository.create()`.
- Create `src/server/routers/interview.ts`:
  - `start` — protected mutation. Input: `{ bookId: string, questionId: string }`. Verifies the book belongs to the user. Delegates to `conversationService.startInterview()`. Returns `{ interviewId, openingMessage }`.
  - `sendMessage` — protected mutation. Input: `{ interviewId: string, content: string }`. Delegates to `conversationService.sendMessage()`. Returns the assistant's response text. Verifies the interview belongs to the user.
  - `getMessages` — protected query. Input: `{ interviewId: string }`. Delegates to `conversationService.getInterviewMessages()`. Verifies the interview belongs to the user.
  - `complete` — protected mutation. Input: `{ interviewId: string }`. Delegates to `conversationService.completeInterview()`. Verifies the interview belongs to the user.
- Update `src/server/routers/_app.ts`:
  - Add `question: questionRouter`, `book: bookRouter`, and `interview: interviewRouter` to the root router
- Create ownership verification helper:
  - `src/server/routers/_helpers.ts` — shared functions to verify resource ownership
  - `verifyBookOwnership(bookId, userId)` — checks book.userId === userId
  - `verifyInterviewOwnership(interviewId, userId)` — follows interview → book → userId chain
  - Used by tRPC routers to enforce access control before delegating to services
- Write tests:
  - `tests/server/routers/question.test.ts` — test list and listByCategory with mock repository
  - `tests/server/routers/book.test.ts` — test getOrCreate, getById, addQuestion with mock repositories. Test UNAUTHORIZED without auth. Test that books from other users are not accessible.
  - `tests/server/routers/interview.test.ts` — test start, sendMessage, getMessages, complete with mock service. Test ownership verification. Test UNAUTHORIZED without auth.

## Notes

- The `book.getOrCreate` router handles the "first-time user" case: if the user has no Book yet, it creates one with a default title. This simplifies the MVP — users don't manage Books directly.
- Per ADR 013, all conversation operations go through tRPC — no separate streaming API route. This gives us full type safety and a single API surface with no duplicated auth.
- The `interview.sendMessage` mutation awaits the full response from the conversation service and returns it. The client sees a loading state while waiting, then gets the complete response. For short interviewer responses this is snappy.
- Ownership verification is important: a user should never be able to access another user's interviews, books, or messages. The chain is always: resource → ... → book → check `book.userId === ctx.userId`.
- Middleware (`src/middleware.ts`) already allows `/api/*` as public routes (auth is enforced at the handler level), so no middleware changes needed.
- The question router is simple — questions are global seed data. No ownership checks needed, just auth.
