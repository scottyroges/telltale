# Plan 1.6: Context Window Management

**Status:** Planned
**Depends on:** 1.2 (needs conversation service), 1.5 (needs thread extraction for thread context injection)
**Blocks:** None in Phase 1

## Goal

Manage the context window for long conversations — summarize older messages, keep recent ones verbatim, inject thread context for coherence. Without this, conversations longer than ~20 messages will hit the context limit or become expensive/slow due to sending the full history every turn.

## Manual Steps (Before Code)

None.

## Key Steps

- Create `src/services/summarization-prompt.ts`:
  - Exports `SUMMARIZATION_PROMPT` as a string constant
  - Instructs Claude to summarize a conversation excerpt
  - Instructions: preserve names, dates, places, emotional moments, key narrative details. Be concise but don't lose specificity. Output a prose summary, not bullet points.
- Create `src/services/context.service.ts`:
  - **`buildContextWindow(sessionId: string)`**
    - Loads all messages for the session via `messageRepository.findBySessionId()`
    - Loads existing threads for the session via `storyThreadRepository.findBySessionId()`
    - Estimates total token count using a heuristic (chars / 4)
    - If total tokens are under the threshold (e.g., 8,000 tokens), return all messages as-is — no summarization needed
    - If total tokens exceed the threshold:
      - Split messages into "old" (to summarize) and "recent" (to keep verbatim, e.g., last 10 messages)
      - Check if the session already has a stored summary (`session.summary`)
      - If no stored summary, or if there are old messages beyond the summarized range: call `summarizeMessages()` to create/update the summary
      - Store the updated summary on the session via `sessionRepository.updateSummary()`
    - Assemble and return the context window:
      1. System prompt (from `prompt.ts`)
      2. If summarized: a SYSTEM message with the summary, framed as "Here is a summary of the earlier conversation: ..."
      3. Thread context: a SYSTEM message listing unexplored threads, framed as "Topics mentioned but not yet explored: ..."
      4. Recent messages verbatim
    - Returns `{ systemPrompt: string, messages: Array<{ role, content }> }` — ready to pass to the Anthropic SDK
  - **`summarizeMessages(messages: Message[])`**
    - Calls `anthropic.messages.create()` with the summarization prompt + messages
    - NOT streaming — returns the full summary text
    - Returns the summary string
  - **Token counting heuristic:**
    - `estimateTokens(text: string): number` — `Math.ceil(text.length / 4)`
    - Simple but effective for English text. No need for a tokenizer library.
    - Constants: `MAX_CONTEXT_TOKENS = 16000`, `SUMMARIZATION_THRESHOLD = 8000`, `RECENT_MESSAGES_COUNT = 10`
- Update `src/services/conversation.service.ts`:
  - `sendMessage` now uses `contextService.buildContextWindow(sessionId)` instead of loading raw message history
  - The context window output goes directly to the Anthropic SDK call
  - The raw user message is still persisted as before — context window assembly doesn't affect persistence
- Write unit tests in `tests/services/context.service.test.ts`:
  - Test: under threshold — returns all messages, no summarization called
  - Test: over threshold — splits messages, calls summarization, stores summary
  - Test: with existing summary — uses stored summary, doesn't re-summarize already-summarized messages
  - Test: thread context injection — includes unexplored threads in the context
  - Test: token estimation heuristic
  - Mock repositories and Anthropic client

## Notes

- The context window strategy is: **summarize old, keep recent, inject threads**. This is a common pattern for long-context chat applications.
- Summarization is **incremental** — we store the summary on the session and only re-summarize when there are new old messages beyond the stored summary. This avoids re-summarizing the entire conversation every turn.
- The threshold constants (8K tokens for summarization trigger, 16K max context, 10 recent messages) are tuned for Claude Sonnet 4.5's 200K context window. We're being conservative — the goal is cost/latency management, not avoiding hard limits.
- Thread context injection serves two purposes: (1) reminds the AI about topics to explore, improving conversation depth, and (2) preserves important details that might be lost during summarization.
- The summarization prompt tells Claude to preserve specificity — "Maria, the sister who lived in Calabria" not "a family member". This is critical because the summary replaces the original messages.
- `buildContextWindow` returns a format ready for the Anthropic SDK — system prompt as a separate field, messages as an array of `{ role, content }` objects. The conversation service just passes this through.
- The SYSTEM messages for summary and threads are injected as the first messages in the array (after the system prompt), before the recent messages. This gives Claude the context before seeing the recent conversation.
- If summarization fails (Claude error), fall back to truncating old messages rather than failing the conversation. Degraded but functional.
