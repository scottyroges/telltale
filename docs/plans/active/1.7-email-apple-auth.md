# Plan 1.7: Email + Apple Sign-In

**Status:** Planned
**Depends on:** 0.4 (Better Auth already configured with Google OAuth)
**Blocks:** Nothing — can be done in parallel with other Phase 1 work

## Goal

Add email/password sign-up and Apple OAuth as additional authentication methods alongside Google OAuth. Users should be able to create an account with email+password, verify their email, reset their password, and sign in with Apple. The login page becomes a multi-method auth page.

## Manual Steps (Before Code)

- `npm install resend` — Email sending library
- `npm install apple-signin-auth` — For Apple JWT client secret generation
- **Apple Developer setup:**
  - Register an App ID with "Sign in with Apple" capability
  - Create a Services ID for web authentication
  - Generate a private key (.p8 file) for Sign in with Apple — note the Key ID shown after creation
  - Download the .p8 file and save its contents
  - Configure the return URL: `{BETTER_AUTH_URL}/api/auth/callback/apple`
- **Email provider setup:**
  - Create account with Resend (https://resend.com)
  - Verify sending domain
  - Get API key
- Add environment variables to `.env`:
  - `APPLE_CLIENT_ID` — Services ID from Apple Developer Console
  - `APPLE_TEAM_ID` — Team ID from Apple Developer Account settings
  - `APPLE_KEY_ID` — Key ID noted when generating the .p8 file
  - `APPLE_PRIVATE_KEY` — Contents of the .p8 file (store as multiline string or base64-encoded)
  - `EMAIL_FROM` — sender address (e.g., `noreply@telltale.app`)
  - `RESEND_API_KEY` — API key from Resend

### Better Auth Configuration

- Update `src/server/auth.ts`:
  - Import Apple JWT generator:
    ```typescript
    import { getClientSecret } from 'apple-signin-auth';
    ```
  - Generate Apple client secret at runtime (before `betterAuth()` call):
    ```typescript
    const appleClientSecret = process.env.APPLE_TEAM_ID
      ? getClientSecret({
          clientId: process.env.APPLE_CLIENT_ID!,
          teamId: process.env.APPLE_TEAM_ID!,
          keyIdentifier: process.env.APPLE_KEY_ID!,
          privateKey: process.env.APPLE_PRIVATE_KEY!,
        })
      : undefined;
    ```
  - Add environment variable checks for Apple credentials (same pattern as Google, lines 7-12):
    ```typescript
    if (!process.env.APPLE_CLIENT_ID) {
      throw new Error("APPLE_CLIENT_ID environment variable is required");
    }
    if (!process.env.APPLE_TEAM_ID) {
      throw new Error("APPLE_TEAM_ID environment variable is required");
    }
    if (!process.env.APPLE_KEY_ID) {
      throw new Error("APPLE_KEY_ID environment variable is required");
    }
    if (!process.env.APPLE_PRIVATE_KEY) {
      throw new Error("APPLE_PRIVATE_KEY environment variable is required");
    }
    ```
  - Enable email/password authentication (top-level option):
    ```typescript
    emailAndPassword: {
      enabled: true,
      requireEmailVerification: true,
      sendResetPassword: async ({ user, url }) => {
        await sendEmail(user.email, "Reset your password", `<a href="${url}">Reset password</a>`);
      },
    },
    ```
  - Add email verification handler (top-level option, separate from emailAndPassword):
    ```typescript
    emailVerification: {
      sendOnSignUp: true,
      autoSignInAfterVerification: true,
      expiresIn: 86400, // 24 hours (default is 3600 = 1 hour)
      sendVerificationEmail: async ({ user, url }) => {
        await sendEmail(user.email, "Verify your email", `<a href="${url}">Verify email</a>`);
      },
    },
    ```
  - Add Apple to `socialProviders`:
    ```typescript
    apple: {
      clientId: process.env.APPLE_CLIENT_ID,
      clientSecret: appleClientSecret,
    },
    ```
  - Keep Google provider as-is

## Notes

- **No schema migration needed.** The existing Better Auth schema already has `Account.password` (nullable), `User.emailVerified`, and the `Verification` model. Better Auth generated these fields in Plan 0.4 — they just weren't used with Google-only OAuth.
- **Email verification is required** for email sign-ups. This prevents fake accounts and ensures we can reach the user. Social sign-ins (Google, Apple) skip verification since the provider already verified the email.
- **Email provider choice:** Resend is recommended for simplicity (single API call, good DX). The `src/lib/email.ts` wrapper keeps the provider swappable — same pattern as the LLM provider in 1.2.
- **Apple Sign-In quirks:** Apple only sends the user's name on the first sign-in. Better Auth handles this, but it means if the first sign-in fails after Apple's redirect, the name is lost. Apple also requires HTTPS for the callback URL, so Apple sign-in won't work in local development without a tunnel (e.g., ngrok). Google works fine on localhost.
- **Apple client secret generation:** We use `apple-signin-auth` to generate the JWT client secret at runtime from the .p8 private key components. This avoids the 6-month expiration issue with pre-generated JWTs. The secret is generated once on server startup, not on every request.
- **Error handling:** Forms use MVP-level error handling — check `error.code` for common cases (e.g., `USER_ALREADY_EXISTS`), show user-friendly messages, fall back to generic message for unknown errors. Client-side validation catches obvious issues (passwords match, email format, password length) before submission.
- **Resend free tier limits:** 100 emails/day, 3,000/month, 2 requests/second. Plenty for MVP testing. No special rate limit handling for now — errors bubble up as generic "Failed to send email". Upgrade to paid plan ($20/mo for 50k emails) when approaching limits in production.
- **Account linking (cross-provider with same email):** Better Auth auto-links by default. If a user signs up with Google, then later signs up with email/password using the same email, Better Auth links the email/password method to the existing user account. Result: one User, multiple sign-in methods. This is standard behavior and good UX.
- **Apple private relay limitation:** If a user signs up with Apple using a private relay email (`abc123@privaterelay.appleid.com`), then later tries to sign up with email/password using their real email, they'll create two separate accounts. This is a known limitation for MVP — account linking only works with matching emails. Rare edge case; can add account merging later if needed.
- **Password security** is handled by Better Auth — it hashes passwords with bcrypt/argon2 internally. No custom hashing needed.
- **"Forgot password" uses generic success message** ("If an account exists...") to prevent email enumeration attacks.
- **Email templates are basic HTML with inline styles.** Simple structure with branding (app name), styled button/link, and proper HTML email formatting. More sophisticated templates (React Email, MJML) can be added later if needed.
- This plan is independent of the Phase 1 interview flow (1.1–1.6) and can be implemented in parallel.
