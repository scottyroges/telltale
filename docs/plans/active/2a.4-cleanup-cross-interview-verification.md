# Plan 2a.4: Core Memory Block — Cleanup + Cross-Interview Verification

**Status:** Not Started
**Created:** 2026-02-26
**Parent plan:** 2a (core memory block)
**Depends on:** 2a.3 (context assembly + conversation service integration)
**Blocks:** 2b (onboarding), 2c (post-interview pipeline)

## Goal

Remove dormant insight code paths from the conversation loop, clean up deprecated types, and verify end-to-end cross-interview memory continuity. After this PR, the codebase has no dead insight code in the conversation/context layer, and we have test coverage proving that memory carries across interviews correctly.

## Why This Is a Separate PR from 2a.3

2a.3 wired core memory into the conversation loop and stopped *creating* insights, but left dormant code in place to keep the diff focused: the `insightRepository` import in `conversationService`, the deprecated `insights` field on `ParsedResponse`, and `ParsedInsight` type references. This PR removes that dead code and adds cross-interview verification tests that prove the full system works end-to-end.

### 1. Remove `insights` Field from `ParsedResponse` (`src/services/response-parser.ts`)

**Current state (after 2a.2):**
```typescript
interface ParsedResponse {

  updatedCoreMemory: string | null;

  parsed: boolean;

}
```

**After:**
```typescript
interface ParsedResponse {
  text: string;
  updatedCoreMemory: string | null;
  shouldComplete: boolean;
  parsed: boolean;
}
```

- Remove the `insights` field from `ParsedResponse`
- Remove the `ParsedInsight` type if it's no longer referenced anywhere
- Remove the `import type { InsightType }` if it was only used by `ParsedInsight`
- Remove the `import { INSIGHT_TYPES }` runtime import if it's no longer used (the validation loop that used it was removed in 2a.2)
- Update all places that construct or destructure `ParsedResponse` to drop `insights`

### 2. Move `getInsights`/`getBookInsights` to Routers, Remove `insightRepository` from `conversationService`

After 2a.3, the `insightRepository` import remains in `conversationService` because `getInsights()` and `getBookInsights()` still delegate to it. These methods are pure pass-throughs — they add no business logic. Move them:

- Remove `getInsights()` and `getBookInsights()` from `conversationService`
- Remove the `insightRepository` import from `conversationService`
- In the interview router (`src/server/routers/interview.ts`), import `insightRepository` directly and call `findByInterviewId`/`findByBookId` in the `getInsights`/`getBookInsights` endpoints

This fully decouples the conversation loop from the insight system. The `insightRepository` itself stays in the codebase — it still has value for the tRPC endpoints that expose historical insight data and will be repurposed in Phase 2c.

### 7. Cross-Interview Verification Tests (`tests/services/conversation.service.test.ts`)

Add focused tests that verify the cross-interview memory lifecycle:

**Test: new interview picks up book memory and resets interview memory**
- Set up a book with `coreMemory` containing both Book Memory and Interview Memory sections (as if a prior interview was in progress or completed — the system makes no distinction)
- Call `startInterview()` for a new interview on the same book
- Assert that `buildContextWindow` was called with memory that has:
  - The Book Memory section intact (same content as before)
  - A fresh `## Interview Memory\nTopic: <new topic>` section (old interview content stripped)
- Note: this also covers the "abandoned interview" case — since no special handling occurs at interview end, the code path is identical whether the prior interview was completed or abandoned

**Test: first interview on a new book works correctly**
- Set up a book with `coreMemory: null`
- Call `startInterview()`
- Assert `buildContextWindow` was called with `null` for core memory
- Assert that after the LLM responds, `bookRepository.updateCoreMemory` is called with the LLM's `updatedCoreMemory` (the book now has memory)

**Test: parse failure preserves existing memory block**
- Set up a book with existing `coreMemory`
- Mock the LLM to return malformed JSON (so parser returns `updatedCoreMemory: null`)
- Call `sendMessage()`
- Assert `bookRepository.updateCoreMemory` was NOT called
- Assert the function still returns successfully (conversation continues)

## Design Decisions

### Insight infrastructure stays in the codebase

The `Insight` model, `insightRepository`, domain types, and tRPC endpoints (`getInsights`, `getBookInsights`) all remain. They serve two purposes: (1) existing books still have historical insight data accessible via the API, and (2) Phase 2c (post-interview pipeline) will repurpose the insight extraction infrastructure for cross-interview indexing and book creation. We're removing insights from the *conversation loop*, not from the codebase.

### Cross-interview tests use mocked repository, not integration tests

These tests verify the conversation service's memory lifecycle logic — stripping interview sections, persisting updates, handling null memory. They use mocked repositories (consistent with existing conversation service tests). Full integration tests with a real database are a separate concern and would belong in an E2E suite.

### `getInsights`/`getBookInsights` move to routers if possible

The conversation service shouldn't depend on `insightRepository` anymore. If the utility methods are just pass-throughs, the tRPC routers can call the repository directly (they're thin wrappers — this is the project convention). This fully decouples the conversation loop from the insight system.

## Known Risks (Out of Scope)

- **`enforceMaxTokens` can drop the core memory message.** Under extreme context pressure, the core memory message could be truncated away, causing the LLM to return garbage `updatedCoreMemory` that overwrites good data. This is tracked as a follow-up from 2a.3, not addressed here.
