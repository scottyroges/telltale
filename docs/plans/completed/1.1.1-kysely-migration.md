# Plan 1.1.1: Migrate Repository Query Layer from Prisma to Kysely

**Status:** Complete
**Depends on:** 1.1 (complete)
**Blocks:** None (repositories maintain the same public API)
**Decision:** ADR 015

## Context

ADR 015 decided on the hybrid approach: keep Prisma for schema/migrations, use Kysely for queries. The 10 repositories from Plan 1.1 currently use the Prisma client for all database operations. This plan migrates them to Kysely while keeping everything else (domain types, services, routers, schema) untouched.

Better Auth supports Kysely natively ŌĆö passing `{ db, type: "postgres" }` shares the application's Kysely instance. This eliminates the `@prisma/client` dependency entirely.

## Key Migration Details

- **Column naming:** DB uses camelCase columns (`userId`, `bookId`, `createdAt`) with snake_case table names (`book_question`, `interview_summary`). No CamelCasePlugin needed.
- **ID generation:** Prisma's `@default(cuid())` runs in JS, not the DB. With Kysely we must generate IDs ourselves using `@paralleldrive/cuid2`.
- **`createdAt`:** Has `DEFAULT CURRENT_TIMESTAMP` at the DB level ŌĆö omit from inserts, get back via `RETURNING`.
- **`updatedAt`:** Prisma's `@updatedAt` is JS-only. We must set `updatedAt: new Date()` manually on every insert and update.
- **Enum defaults:** DB-level defaults exist but we continue setting them explicitly for clarity.
- **`executeTakeFirst()`** returns `undefined`, not `null`. Add `?? null` for `findById`-style methods.
- **`findByIdWithDetails`:** Uses `jsonArrayFrom`/`jsonObjectFrom` from `kysely/helpers/postgres` for nested relations.

## Key Steps

### Step 1: Install Dependencies

```bash
npm install kysely pg @paralleldrive/cuid2
npm install -D prisma-kysely @types/pg
```

### Step 2: Add prisma-kysely Generator + Generate Types

**File:** `prisma/schema.prisma` ŌĆö add generator block:

```prisma
generator kysely {
  provider     = "prisma-kysely"
  output       = "../src/db"
  fileName     = "types.ts"
  enumFileName = "enums.ts"
}
```

Run `npx prisma generate`. Produces:
- `src/db/types.ts` ŌĆö `DB` interface with tableŌåÆtype mappings
- `src/db/enums.ts` ŌĆö runtime enum constants

### Step 3: Create Kysely Database Instance

**New file:** `src/lib/db.ts`

- `import "server-only"` guard
- `Kysely<DB>` singleton with `PostgresDialect` using `pg.Pool`
- Connection string from `DATABASE_URL`
- Same global singleton pattern as `prisma.ts`

### Step 4: Create ID Generation Utility

**New file:** `src/lib/id.ts`

- Re-export `createId` from `@paralleldrive/cuid2`
- Single import point for all repositories

### Step 5: Rewrite 10 Repository Files

Replace `import { prisma } from "@/lib/prisma"` with `import { db } from "@/lib/db"` and `import { createId } from "@/lib/id"`.

**Prisma ŌåÆ Kysely method mapping:**

| Prisma | Kysely |
|--------|--------|
| `prisma.model.create({ data, select })` | `db.insertInto("table").values({id: createId(), ...}).returning([...]).executeTakeFirstOrThrow()` |
| `prisma.model.findUnique({ where: {id}, select })` | `db.selectFrom("table").where("id", "=", id).select([...]).executeTakeFirst()` then `?? null` |
| `prisma.model.findMany({ where, select, orderBy })` | `db.selectFrom("table").where(...).select([...]).orderBy(...).execute()` |
| `prisma.model.findFirst({ where, orderBy })` | `db.selectFrom("table").where(...).select([...]).orderBy(...).limit(1).executeTakeFirst()` then `?? null` |
| `prisma.model.update({ where, data, select })` | `db.updateTable("table").set({..., updatedAt: new Date()}).where(...).returning([...]).executeTakeFirstOrThrow()` |
| `prisma.model.delete({ where, select })` | `db.deleteFrom("table").where(...).returning([...]).executeTakeFirstOrThrow()` |
| `prisma.model.createMany({ data })` | `db.insertInto("table").values([...]).execute()` ŌĆö return `{ count: data.length }` |

**Special: `book.findByIdWithDetails`** ŌĆö use `jsonArrayFrom` + `jsonObjectFrom` from `kysely/helpers/postgres` for bookQuestions (with nested question) and interviews via lateral subqueries.

**Files:** `user.repository.ts`, `question.repository.ts`, `book.repository.ts`, `book-question.repository.ts`, `interview.repository.ts`, `message.repository.ts`, `insight.repository.ts`, `interview-summary.repository.ts`, `story.repository.ts`, `story-section.repository.ts`

### Step 6: Rewrite 9 Repository Test Files

Mock `@/lib/db` instead of `@/lib/prisma`. Also mock `@/lib/id` to return deterministic IDs.

Create a test helper (`tests/helpers/mock-db.ts`) that builds chainable mock objects for Kysely's fluent API. Terminal methods (`execute`, `executeTakeFirst`, `executeTakeFirstOrThrow`) are `vi.fn()` that return test data.

Tests verify:
1. The correct table is targeted (e.g., `selectFrom` called with `"book"`)
2. The terminal method returns the expected domain type
3. Core arguments (where clauses, etc.) where practical

### Step 7: Verify

- `npm test` ŌĆö all tests pass
- `npm run build` ŌĆö no type errors
- `npm run lint` ŌĆö passes
- Manual: start dev server, verify Better Auth login/logout still works

## Notes

- `@prisma/client` fully removed from `dependencies`. Better Auth uses the shared Kysely instance via `{ db, type: "postgres" }`.
- `prisma-kysely` types regenerate on every `npx prisma generate` alongside the Prisma client.
- `src/db/` generated files should be committed (they're small and avoid requiring `prisma generate` before builds).
- `createMany` for insights: Kysely throws on insert failure, so returning `{ count: data.length }` is safe.
- Message and InterviewSummary have no `updatedAt` ŌĆö their inserts skip that field entirely.
