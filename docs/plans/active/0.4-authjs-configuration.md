# Plan 0.4: Auth.js Configuration

**Status:** Not Started
**Depends on:** 0.3 (need Prisma schema with Auth.js models and working database)
**Blocks:** 0.5, 0.6

## Goal

Auth.js v5 fully configured with Prisma adapter, Google OAuth, sign-in/sign-out pages, middleware protecting the `(app)` route group, and session access server-side via `auth()`.

## Manual Steps (Before Code)

1. **Create Google OAuth credentials:**
   - Go to https://console.cloud.google.com
   - Create project "Telltale" (or use existing)
   - APIs & Services → Credentials → Create OAuth client ID
   - Type: Web application
   - Authorized JS origins: `http://localhost:3000`
   - Authorized redirect URIs: `http://localhost:3000/api/auth/callback/google`
   - Copy Client ID and Client Secret

2. **Generate AUTH_SECRET:**
   - Run `npx auth secret` or `openssl rand -base64 33`

3. **Update `.env`:**
   - `AUTH_SECRET="<generated>"`
   - `AUTH_GOOGLE_ID="<client-id>"`
   - `AUTH_GOOGLE_SECRET="<client-secret>"`

## Key Steps

- Install `next-auth@^5.0.0`, `@auth/prisma-adapter`
- Create `src/server/auth.ts`:
  - NextAuth with Prisma adapter, Google provider
  - Custom `signIn` page at `/login`
  - JWT session strategy (`strategy: "jwt"`) — avoids DB round-trip on every request and avoids Edge Runtime issues with Prisma in middleware
  - `jwt` callback attaching `user.id` to the token
  - `session` callback attaching `user.id` from the token to the session (for tRPC context later)
- Create `src/app/api/auth/[...nextauth]/route.ts` — export `GET`/`POST` from handlers
- Create `src/middleware.ts`:
  - Use `auth` as middleware wrapper (JWT strategy allows this without DB access)
  - Allow public routes: `/`, `/api/auth/*`
  - Redirect unauthenticated users to `/login`
  - Redirect authenticated users away from `/login` to `/dashboard`
  - Export `config.matcher` excluding `_next/static`, `_next/image`, `favicon.ico`
- Create `src/app/(auth)/layout.tsx` — minimal auth layout
- Create `src/app/(auth)/login/page.tsx` — Server Component with Google sign-in button via Server Action (`signIn("google")`)
- Create `src/app/(app)/layout.tsx` — calls `auth()` server-side, renders header with user name + sign out button
- Create `src/app/(app)/dashboard/page.tsx` — placeholder showing welcome message
- Root layout (`src/app/layout.tsx`) — no `SessionProvider` needed in this plan (deferred to Plan 0.5)

## Testing

Backend tests live in `tests/` mirroring `src/` structure (per ADR 010).

**Unit tests (`tests/server/auth.test.ts`):**
- Auth config exports `auth`, `signIn`, `signOut`, `handlers`
- JWT callback attaches `user.id` to token
- Session callback attaches `user.id` from token to session

**Unit tests (`tests/middleware.test.ts`):**
- Unauthenticated request to `/dashboard` redirects to `/login`
- Unauthenticated request to `/` is allowed through
- Unauthenticated request to `/api/auth/callback/google` is allowed through
- Authenticated request to `/login` redirects to `/dashboard`
- Authenticated request to `/dashboard` is allowed through

**E2E tests (`e2e/auth.spec.ts`):**
- Full OAuth flow is deferred — requires real Google credentials and is better suited for manual verification or CI with test credentials
- E2E smoke test: visiting `/dashboard` without auth redirects to `/login`
- E2E smoke test: `/login` page renders with Google sign-in button

## Notes

- Email provider omitted for Phase 0 — Google OAuth is sufficient to prove auth works
- Auth.js v5 uses `AUTH_` env var prefix (auto-detected)
- JWT strategy chosen over database sessions to avoid Edge Runtime compatibility issues with Prisma and to eliminate per-request DB queries in middleware
- Prisma adapter still stores User and Account records in the database — JWT only affects session storage
- `SessionProvider` deferred to Plan 0.5 — this plan uses `auth()` server-side exclusively
