# Plan 0.6: Vercel Deployment

**Status:** Complete
**Depends on:** 0.5 (need the full working application)
**Blocks:** Nothing — this completes Phase 0

## Goal

Deploy to Vercel with environment variables, automatic deploys on push to `main`, preview deploys on PRs, and automated production database migrations.

## Decisions

- **Turbopack:** Dev only, drop from production build (see ADR 012)
- **Database:** All environments share the production Neon database for now; add Neon branching later if needed
- **`BETTER_AUTH_SECRET`:** Generate a unique secret for production (do not reuse local)
- **`BETTER_AUTH_URL`:** Set for production; preview deploys fall back to `VERCEL_URL`
- **`trustedOrigins`:** Configure in auth config for both production and preview URLs
- **Migrations:** `prisma migrate deploy` runs on all builds (acceptable risk — solo developer, shared DB)

## Part 1: Manual Steps (You Do These)

### Step 1: Provision Neon Database

1. Go to https://neon.tech and create a free account (use GitHub login)
2. Create a new project named **"telltale"**
3. Select **AWS us-east-1** (closest to Vercel's default region)
4. Once created, go to **Dashboard → Connection Details**
5. Copy two connection strings:
   - **Pooled connection string** → this becomes `DATABASE_URL`
     - Looks like: `postgresql://user:pass@ep-xxx.us-east-1.aws.neon.tech/telltale?sslmode=verify-full`
     - Make sure "Pooled connection" is toggled **on**
     - **Important:** Use `sslmode=verify-full` (not `require`) to prevent pg v9 security downgrade
   - **Direct connection string** → this becomes `DIRECT_URL`
     - Same but with `-pooler` removed from the hostname
     - Also use `?sslmode=verify-full`
     - Used by Prisma for migrations (DDL statements need direct connections)
6. **Test locally** (optional but recommended):
   - Temporarily swap your `.env` values to the Neon strings
   - Run `npx prisma migrate deploy` — should apply all migrations
   - Run `npm run dev` and verify the app works
   - Swap back to your local Docker DB strings when done

### Step 2: Create Vercel Account & Link Project

1. Go to https://vercel.com and sign up with your **GitHub account** (free tier)
2. Install the Vercel CLI:
   ```
   npm install -g vercel
   ```
3. Log in:
   ```
   vercel login
   ```
4. Link the project:
   ```
   vercel link
   ```
   - Select **"Create new project"**
   - Name it **"telltale"**
   - Accept the default settings (Vercel auto-detects Next.js)
5. **Important:** Go to the Vercel dashboard → **telltale** project → **Settings → Git**
   - Connect your GitHub repository (this enables auto-deploys on push to `main` and preview deploys on PRs)
   - `vercel link` alone does NOT set up git-push deploys — this step is required

### Step 3: Set Environment Variables in Vercel

Go to Vercel dashboard → **telltale** → **Settings → Environment Variables**.

Set these for **all environments** (Production, Preview, Development):

| Variable | Value | Notes |
|---|---|---|
| `DATABASE_URL` | Neon pooled connection string | From Step 1 |
| `DIRECT_URL` | Neon direct connection string | From Step 1 |
| `GOOGLE_CLIENT_ID` | Your Google OAuth client ID | Same as local `.env` |
| `GOOGLE_CLIENT_SECRET` | Your Google OAuth client secret | Same as local `.env` |

Set this for **Production only**:

| Variable | Value | Notes |
|---|---|---|
| `BETTER_AUTH_URL` | `https://telltale-<slug>.vercel.app` | Your actual production URL (visible after first deploy) |

Set this for **all environments** — generate a **new** secret, do NOT reuse your local one:

| Variable | Value | Notes |
|---|---|---|
| `BETTER_AUTH_SECRET` | Run `openssl rand -base64 32` | Unique production secret |

> **Note:** `BETTER_AUTH_URL` is only set for production. Preview deploys will auto-detect their URL via `VERCEL_URL` (handled in the code change below). You may need to come back and set `BETTER_AUTH_URL` after the first deploy once you know the actual URL.

### Step 4: Update Google OAuth Redirect URIs

1. Go to https://console.cloud.google.com → **APIs & Services → Credentials**
2. Click on your OAuth 2.0 Client ID
3. Under **Authorized redirect URIs**, add:
   ```
   https://telltale-<slug>.vercel.app/api/auth/callback/google
   ```
   (Replace `<slug>` with your actual Vercel subdomain)
4. Under **Authorized JavaScript origins**, add:
   ```
   https://telltale-<slug>.vercel.app
   ```
5. Click **Save**

> **Note:** Preview deploys will NOT have Google OAuth working unless you add each preview URL (impractical). Auth on preview deploys is best-effort — production is what matters.

### Step 5: Verify Build Locally

Before deploying, confirm the build works:
```
npm run build
```
This should run `prisma migrate deploy && next build` successfully (it will migrate your local DB and build the app).

## Part 2: Code Changes (Claude Does These)

### 2a. Update build script in `package.json`

```diff
- "build": "next build --turbopack",
+ "build": "prisma migrate deploy && next build",
```

- Drops `--turbopack` from production builds (ADR 012)
- `prisma migrate deploy` applies pending migrations before each build (idempotent, safe)
- `prisma migrate deploy` uses `DIRECT_URL` automatically (bypasses connection pooler for DDL)

2. Configure `trustedOrigins` to accept requests from the resolved base URL
