# Plan 2b.3: Onboarding Service + tRPC Router

**Status:** Not Started
**Created:** 2026-03-02
**Parent plan:** 2b (onboarding + interview planning)
**Depends on:** 2b.2 (plan generation service + prompt)
**Blocks:** 2b.4 (onboarding UI)

## Goal

Add a thin onboarding orchestration service and a tRPC router that exposes it. The service coordinates life area response persistence, onboarding status transitions, and plan generation. The router handles auth, ownership verification, and input validation. No UI yet — testable via unit tests and router tests.

After this PR, the frontend has everything it needs to build the onboarding form: `onboarding.getStatus`, `onboarding.saveResponse`, `onboarding.deleteResponse`, and `onboarding.complete`.

## Changes

### 1. Onboarding Service

**`src/services/onboarding.service.ts`** — new file. Thin orchestration — no business logic beyond validation and delegation.

```typescript
import { lifeAreaResponseRepository } from "@/repositories/life-area-response.repository";
import { interviewTopicRepository } from "@/repositories/interview-topic.repository";
import { bookRepository } from "@/repositories/book.repository";
import { planGenerationService } from "@/services/plan-generation.service";
import { LIFE_AREAS } from "@/domain/life-area";
import type { LifeAreaResponse } from "@/domain/life-area";
import type { InterviewTopic } from "@/domain/interview-topic";
import type { OnboardingStatus } from "@/domain/book";

const LIFE_AREA_KEY_SET = new Set(LIFE_AREAS.map((a) => a.key));
const MIN_RESPONSES = 2;
```

Methods:

**`saveResponse(bookId, lifeAreaKey, content)`:**

```typescript
async saveResponse(
  bookId: string,
  lifeAreaKey: string,
  content: string,
): Promise<LifeAreaResponse> {
  if (!LIFE_AREA_KEY_SET.has(lifeAreaKey as any)) {
    throw new Error(`Invalid life area key: ${lifeAreaKey}`);
  }

  const book = await bookRepository.findById(bookId);
  if (!book) throw new Error(`Book not found: ${bookId}`);

  if (book.onboardingStatus === "NOT_STARTED") {
    await bookRepository.updateOnboardingStatus(bookId, "IN_PROGRESS");
  }

  return lifeAreaResponseRepository.upsert(bookId, lifeAreaKey, content);
},
```

- Validates `lifeAreaKey` against `LIFE_AREAS`
- Transitions `NOT_STARTED` → `IN_PROGRESS` on first save
- Upserts (idempotent — saving the same area again replaces the content)
- Does NOT transition if already `IN_PROGRESS` or `COMPLETE`

**`deleteResponse(bookId, lifeAreaKey)`:**

```typescript
async deleteResponse(bookId: string, lifeAreaKey: string): Promise<void> {
  await lifeAreaResponseRepository.deleteByBookIdAndKey(bookId, lifeAreaKey);
},
```

- Simple passthrough — no status transition needed (area can be re-added later)

**`getStatus(bookId)`:**

```typescript
async getStatus(bookId: string): Promise<{
  onboardingStatus: OnboardingStatus;
  responses: LifeAreaResponse[];
  topics: InterviewTopic[];
}> {
  const book = await bookRepository.findById(bookId);
  if (!book) throw new Error(`Book not found: ${bookId}`);

  const [responses, topics] = await Promise.all([
    lifeAreaResponseRepository.findByBookId(bookId),
    interviewTopicRepository.findByBookId(bookId),
  ]);

  return {
    onboardingStatus: book.onboardingStatus,
    responses,
    topics,
  };
},
```

- Returns everything the UI needs in one call
- Loads responses and topics in parallel
- Topics will be empty until `completeOnboarding` is called

**`completeOnboarding(bookId)`:**

```typescript
async completeOnboarding(bookId: string): Promise<InterviewTopic[]> {
  const responses = await lifeAreaResponseRepository.findByBookId(bookId);

  if (responses.length < MIN_RESPONSES) {
    throw new Error(
      `At least ${MIN_RESPONSES} life area responses required, got ${responses.length}`,
    );
  }

  return planGenerationService.generatePlan(bookId);
},
```

- Validates minimum response count (at least 2 life areas answered)
- Delegates to `planGenerationService.generatePlan()` which handles topic creation, core memory bootstrap, and setting `onboardingStatus` to `COMPLETE`
- Returns the generated topics

### 2. tRPC Router

**`src/server/routers/onboarding.ts`** — new file:

```typescript
import { z } from "zod";
import { router, approvedProcedure } from "@/server/trpc";
import { verifyBookOwnership } from "@/server/routers/ownership";
import { onboardingService } from "@/services/onboarding.service";

export const onboardingRouter = router({
  getStatus: approvedProcedure
    .input(z.object({ bookId: z.string() }))
    .query(async ({ ctx, input }) => {
      await verifyBookOwnership(input.bookId, ctx.userId);
      return onboardingService.getStatus(input.bookId);
    }),

  saveResponse: approvedProcedure
    .input(
      z.object({
        bookId: z.string(),
        lifeAreaKey: z.string(),
        content: z.string().min(1).max(5000),
      }),
    )
    .mutation(async ({ ctx, input }) => {
      await verifyBookOwnership(input.bookId, ctx.userId);
      return onboardingService.saveResponse(
        input.bookId,
        input.lifeAreaKey,
        input.content,
      );
    }),

  deleteResponse: approvedProcedure
    .input(
      z.object({
        bookId: z.string(),
        lifeAreaKey: z.string(),
      }),
    )
    .mutation(async ({ ctx, input }) => {
      await verifyBookOwnership(input.bookId, ctx.userId);
      return onboardingService.deleteResponse(input.bookId, input.lifeAreaKey);
    }),

  complete: approvedProcedure
    .input(z.object({ bookId: z.string() }))
    .mutation(async ({ ctx, input }) => {
      await verifyBookOwnership(input.bookId, ctx.userId);
      return onboardingService.completeOnboarding(input.bookId);
    }),
});
```

All procedures use `approvedProcedure` (matching the interview router pattern — onboarding is a core feature requiring an approved account). Every procedure verifies book ownership via the existing `verifyBookOwnership` helper.

The router is thin — input validation, auth, ownership check, then delegate to service. No business logic.

- Does NOT transition when already `COMPLETE` (`updateOnboardingStatus` not called)

- Calls `deleteByBookIdAndKey` with correct args

- Returns `{ onboardingStatus, responses, topics }` from parallel loads
- Throws when book not found
- Returns empty arrays for responses and topics when none exist

**`completeOnboarding`:**
- Throws when fewer than 2 responses exist
- Calls `planGenerationService.generatePlan(bookId)` when validation passes
- Returns generated topics from `generatePlan`
