# Plan 1.7: Email Authentication

**Status:** Complete (PR 1 ✅, PR 2 ✅, PR 3 ✅)
**Depends on:** 0.4 (Better Auth already configured with Google OAuth)
**Blocks:** Nothing — can be done in parallel with other Phase 1 work

## Goal

Add email/password sign-up as an additional authentication method alongside Google OAuth. Users should be able to create an account with email+password, verify their email, reset their password, and sign in with either method. The login page becomes a multi-method auth page.

**Note:** Apple Sign-In was originally planned but has been removed from scope.

## Manual Steps (Before Code)

- `npm install resend` — Email sending library
- **Email provider setup:**
  - Create account with Resend (https://resend.com)
  - Verify sending domain
  - Get API key
- Add environment variables to `.env`:
  - `EMAIL_FROM` — sender address (e.g., `noreply@telltale.app`)
  - `RESEND_API_KEY` — API key from Resend

## Key Steps

### Email Provider Setup

- Create `src/lib/email.ts`:
  - `import "server-only"` guard
  - Exports a `sendEmail` function: `(to: string, subject: string, html: string) => Promise<void>`
  - Implements with Resend SDK
  - Reads `RESEND_API_KEY` and `EMAIL_FROM` from environment
  - **Error handling:** Throw if Resend returns an error — this blocks the auth operation and gives immediate feedback to the user rather than silent failure
  - Environment variable validation: throw on startup if `RESEND_API_KEY` or `EMAIL_FROM` is missing
  - This is the only file that knows which email provider is active — same isolation principle as the LLM provider
- Create basic HTML email templates (inline styles, max-width 600px):
  - Simple structure: header with app name, content paragraph, styled button/link, footer

### Better Auth Configuration

- Update `src/server/auth.ts`:
  - Enable email/password authentication (top-level option):
    ```typescript
    emailAndPassword: {
      enabled: true,
      requireEmailVerification: true,
      sendResetPassword: async ({ user, url }) => {
        await sendEmail(user.email, "Reset your password", `<a href="${url}">Reset password</a>`);
      },
    },
    ```
  - Add email verification handler (top-level option, separate from emailAndPassword):
    ```typescript
    emailVerification: {
      sendOnSignUp: true,
      autoSignInAfterVerification: true,
      expiresIn: 86400, // 24 hours (default is 3600 = 1 hour)
      sendVerificationEmail: async ({ user, url }) => {
        await sendEmail(user.email, "Verify your email", `<a href="${url}">Verify email</a>`);
      },
    },
    ```
  - Keep Google provider as-is

## Notes

- **No schema migration needed.** The existing Better Auth schema already has `Account.password` (nullable), `User.emailVerified`, and the `Verification` model. Better Auth generated these fields in Plan 0.4 — they just weren't used with Google-only OAuth.
- **Email verification is required** for email sign-ups. This prevents fake accounts and ensures we can reach the user. Social sign-ins (Google) skip verification since the provider already verified the email.
- **Email provider choice:** Resend is recommended for simplicity (single API call, good DX). The `src/lib/email.ts` wrapper keeps the provider swappable — same pattern as the LLM provider in 1.2.
- **Error handling:** Forms use MVP-level error handling — check `error.code` for common cases (e.g., `USER_ALREADY_EXISTS`), show user-friendly messages, fall back to generic message for unknown errors. Client-side validation catches obvious issues (passwords match, email format, password length) before submission.
- **Resend free tier limits:** 100 emails/day, 3,000/month, 2 requests/second. Plenty for MVP testing. No special rate limit handling for now — errors bubble up as generic "Failed to send email". Upgrade to paid plan ($20/mo for 50k emails) when approaching limits in production.
- **Account linking (cross-provider with same email):** Better Auth auto-links by default. If a user signs up with Google, then later signs up with email/password using the same email, Better Auth links the email/password method to the existing user account. Result: one User, multiple sign-in methods. This is standard behavior and good UX.
- **Password security** is handled by Better Auth — it hashes passwords with bcrypt/argon2 internally. No custom hashing needed.
- **"Forgot password" uses generic success message** ("If an account exists...") to prevent email enumeration attacks.
- **Email templates are basic HTML with inline styles.** Simple structure with branding (app name), styled button/link, and proper HTML email formatting. More sophisticated templates (React Email, MJML) can be added later if needed.
- This plan is independent of the Phase 1 interview flow (1.1–1.6) and can be implemented in parallel.
