# Plan 2a.3: Core Memory Block — Context Assembly + Conversation Service Integration

**Status:** Complete
**Created:** 2026-02-26
**Parent plan:** 2a (core memory block)
**Depends on:** 2a.2 (system prompt + response format)
**Blocks:** 2a.4 (cleanup + cross-interview verification)

## Goal

Wire the core memory block into the live conversation loop end-to-end. After this PR, the AI interviewer reads its accumulated memory before each turn, returns an updated memory block in every response, and the system persists it to the Book. Cross-interview continuity works: interview #2 picks up book-level memory from interview #1, with a fresh interview section.

## Why This Is a Separate PR from 2a.2

2a.2 changed the prompt and parser — the LLM now returns `updatedCoreMemory` instead of `insights[]`, and the parser extracts it. But nothing *reads* or *writes* core memory yet. The conversation service still calls `insightRepository.createMany()` (which writes nothing since `insights` is always `[]`), and the context service still loads insights (which returns an empty list since nothing creates them anymore).

This PR connects the plumbing: the context service injects core memory into the context window, the conversation service persists updated memory to the Book, and new interviews get a fresh interview section while preserving book memory.

## Changes

### 1. Context Service Signature Change (`src/services/context.service.ts`)

**Current signature:**
```typescript
async buildContextWindow(interviewId: string, userName: string): Promise<ContextWindow>
```

**New signature:**
```typescript
async buildContextWindow(interviewId: string, userName: string, coreMemory: string | null): Promise<ContextWindow>
```

The caller passes the memory string directly. This lets `startInterview` pass stripped memory (book section only + fresh interview header) while `sendMessage`/`redirect` pass the raw book value. The context service doesn't need to know about memory lifecycle — it just injects what it's given.

### 2. Replace Insight Injection with Core Memory Injection (`src/services/context.service.ts`)

**Remove:**
- `import { insightRepository }` — no longer needed
- `import type { Insight }` — no longer needed
- `buildInsightContextMessage()` function
- `assembleMessagesWithInsights()` function
- Insight loading in `buildContextWindow()` (`insightRepository.findByInterviewId(interviewId)`)
- `insights` parameter from all assembly functions (`assembleUnderThreshold`, `assembleSummarized`, `assembleIncremental`)
- `insights` field from `TokenBreakdown` interface

**Add:**
- `buildCoreMemoryContextMessage(coreMemory: string): string` — wraps the raw memory block: `"[Your memory — what you know about this subject]\n" + coreMemory`
- `assembleMessagesWithCoreMemory(messages: Message[], coreMemory: string | null): LLMMessage[]` — same injection pattern as `assembleMessagesWithInsights` but with core memory: injects as an assistant-role message before the last user message. When `coreMemory` is `null`, no injection (same as empty insights today).
- `coreMemory: number` field on `TokenBreakdown` (replaces `insights: number`)
- `calculateTokenBreakdown()` accepts `coreMemory: string | null` parameter instead of `insights: Insight[]`

**Assembly function signatures change:**
- `assembleUnderThreshold(messages, coreMemory, systemPrompt)` — `Insight[]` → `string | null`
- `assembleSummarized(interviewId, messages, buckets, coreMemory, existingSummary, systemPrompt)` — same
- `assembleIncremental(buckets, coreMemory, existingSummary, systemPrompt)` — same

All three call `assembleMessagesWithCoreMemory()` instead of `assembleMessagesWithInsights()`.

**`buildContextWindow()` body changes:**
- The `Promise.all` drops `insightRepository.findByInterviewId(interviewId)` — three parallel loads instead of four
- `calculateTokenBreakdown()` receives `coreMemory` (the string param) instead of `insights`
- All assembly calls pass `coreMemory` instead of `insights`

### 3. Conversation Service — Add Book Repository + Memory Persistence (`src/services/conversation.service.ts`)

**New import:**
```typescript
import { bookRepository } from "@/repositories/book.repository";
```

**Remove:**
- All `insightRepository.createMany()` calls (three total: `startInterview`, `sendMessage`, `redirect`)

Note: The `insightRepository` import stays — `getInsights` and `getBookInsights` still use it. Cleaned up in Phase 4.

**`startInterview(bookId, topic, userName)` changes:**
1. After creating the interview and hidden topic message, load the book's current core memory: `const book = await bookRepository.findById(bookId)`
2. Strip the interview section from `book.coreMemory` (if it exists):
   - Find the `## Interview Memory` marker
   - Keep everything before it
   - Append a fresh `## Interview Memory\nTopic: ${topic}` header
   - If `coreMemory` is `null` (first interview on this book), pass `null` to context service
3. Pass the prepared memory to context service: `contextService.buildContextWindow(interview.id, userName, preparedMemory)`
4. After parsing the response, if `parsed.updatedCoreMemory` is not `null`, persist: `await bookRepository.updateCoreMemory(bookId, parsed.updatedCoreMemory)`

**`sendMessage(interviewId, bookId, content, userName)` changes:**
1. Save the user message and load the book in parallel: `Promise.all([messageRepository.create(...), bookRepository.findById(bookId)])`
2. Pass `book?.coreMemory ?? null` to context service: `contextService.buildContextWindow(interviewId, userName, book?.coreMemory ?? null)`
3. After parsing, if `parsed.updatedCoreMemory` is not `null`, persist: `await bookRepository.updateCoreMemory(bookId, parsed.updatedCoreMemory)`
4. Remove `insightRepository.createMany()` call

**`redirect(interviewId, bookId, userName)` changes:**
- Same pattern as `sendMessage`: save hidden redirect message and load book in parallel, pass core memory to context service, persist updated memory, remove insight creation.

### 4. Interview Section Stripping Helper

Extract the interview section stripping logic into a small pure function in `conversation.service.ts` (internal, not exported):

```typescript
function prepareMemoryForNewInterview(
  coreMemory: string | null,
  topic: string,
): string | null {
  if (!coreMemory) return null;

  const marker = "## Interview Memory";
  const markerIndex = coreMemory.indexOf(marker);

  const bookSection = markerIndex >= 0
    ? coreMemory.substring(0, markerIndex).trimEnd()
    : coreMemory.trimEnd();

  return `${bookSection}\n\n## Interview Memory\nTopic: ${topic}`;
}
```

This is a deterministic string operation. No LLM call needed.

## Design Decisions

### Callers pass core memory to context service, not the context service loading it

The parent plan specifies this and it's the right call. The context service is a pure assembly function — it doesn't own memory lifecycle decisions. The conversation service decides *which* memory to pass (raw for `sendMessage`, stripped for `startInterview`). This keeps the context service simple and testable.

### Interview section stripping lives in conversation service

The conversation service owns the lifecycle decision of when a new interview resets session memory. The stripping is a simple string operation — find `## Interview Memory`, keep everything before it, append a fresh header with the new topic. This is deterministic, no LLM call needed.

### `getInsights` and `getBookInsights` stay for now

These conversation service methods still delegate to `insightRepository`. They're used by tRPC endpoints that return historical data. They'll be cleaned up in Phase 4 (cleanup) — this PR focuses on the conversation loop, not endpoint removal.

### `enforceMaxTokens` can drop the core memory message

The core memory injection sits before the last user message. In extreme truncation (e.g., a very large recent user message), `enforceMaxTokens` could drop it. This is inherited behavior — insights had the same vulnerability — but the impact is higher now that core memory is the AI's primary working memory. TODO: consider pinning the memory message so it's never truncated (separate task, not this PR).

### Parse failure = don't overwrite memory

When `parsed.updatedCoreMemory` is `null` (parse failure), the conversation service does NOT call `bookRepository.updateCoreMemory()`. The existing memory on the Book is preserved. One turn's memory update is lost, but all prior accumulated memory survives. This is the behavior specified in the parent plan.
